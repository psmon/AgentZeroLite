# v1.2.0 — Unit-test trigger policy + dedicated test-runner agent

**Released**: 2026-05-02
**Bump rationale**: Minor — new agent (`test-runner`), new convention document
(`unit-test-policy.md`), and behavioral changes to two existing agents and one
engine. No breaking changes to existing logs, but the *expected behavior* of
release builds and post-edit workflows changes.

## What changed

### New

- **`harness/agents/test-runner.md`** — single owner of `dotnet test` invocation.
  Three modes:
  - **Scoped**: `연관된 유닛테스트 수행해` → git-diff-driven `--filter`
  - **Full**: `전체 유닛테스트 수행해` → headless suite (and AgentTest if
    desktop session)
  - **History**: `유닛테스트 이력` / `최근 유닛테스트 이력알려줘` → read-only
    summary of recent test-runner logs
  - **No auto-trigger.** Runs only on these explicit user phrases.

- **`harness/knowledge/unit-test-policy.md`** — the binding rule:
  > Unit tests are never auto-triggered. They run only when the user
  > explicitly asks via test-runner's triggers.

### Changed behavior

- **`harness/agents/build-doctor.md`** — no longer runs `dotnet test`. Build
  pipeline audit only (`dotnet build -c Debug`, version.txt, native DLLs,
  EF migrations, csproj configs).
- **`harness/agents/test-sentinel.md`** — no longer executes tests. Becomes a
  pure structural auditor (boundary integrity, hot-spot coverage, suite-health
  citation from test-runner logs). Still owns `dotnet-test-execution.md` canon
  for test-runner to follow.
- **`harness/engine/release-build-pipeline.md`** — Step 2 (build-doctor) no
  longer includes a test phase. Security gate + build only. Tests, if desired
  before a release, must be invoked separately by the user via the test-runner
  triggers.

### Unchanged

- `pre-commit-review` engine and `code-coach` Mode 2 — still run on commits
  with dev-code diffs. They review code; they do not run tests.
- `dotnet build` after code edits — still happens automatically to catch
  compile errors. Build is fast and cheap; tests are slow and explicit.

## Why

LLM-backed smoke tests in this repo (Whisper, Gemma, Webnori) are slow —
running them after every change burns RAM (see the 12 GB testhost incident in
`harness/knowledge/dotnet-test-execution.md`) for marginal signal value. The
user has chosen to run tests deliberately and to have logs that record *why*
each run happened, so future improvement passes can mine "what triggered the
last failure" rather than "what auto-fired routinely".

## Migration notes

If a previous habit / external automation depended on:
- "build-doctor runs tests" — invoke `test-runner` separately after build.
- "release pipeline includes a test gate" — invoke `전체 유닛테스트 수행해`
  before the release trigger.
- "test-sentinel runs the suite" — Sentinel now reads test-runner logs
  instead. If the latest log is stale (> 7 days) Sentinel flags it.
