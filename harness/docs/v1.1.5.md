# v1.1.5 — dotnet test execution canon

**Date**: 2026-04-28
**Bump**: patch (knowledge addition + agent procedure refinement, no count change)

## What changed

### New canon document

- **`harness/knowledge/dotnet-test-execution.md`** — single source of truth for how
  agents must invoke `dotnet test`. Codifies six rules (R1–R6) and a host-cleanup
  checklist. Includes the 2026-04-28 incident as the motivating sample.

### Agent procedure refinements

- **`harness/agents/test-sentinel.md`** — Procedure section gains a top-of-section
  canon banner pointing at the new knowledge doc. Steps 1, 2, and 6 now cite
  specific canon rules (R1 single foreground call, R2 empty-output handling, R4
  `--filter` for diagnostic loops, R5 desktop-only WPF suite, R6 testhost-cleanup
  check before reporting). New evaluation axis: **Test-execution hygiene** (Pass/Fail).

- **`harness/agents/build-doctor.md`** — Mirrors the canon banner; Steps 2, 3, 6
  cite canon R1/R3/R5/R6. New evaluation axis: **Test-execution hygiene** (Pass/Fail).
  The Doctor inherits the rule but the Sentinel owns it.

### Engine refinement

- **`harness/engine/release-build-pipeline.md`** — Step 2 (build-doctor) explicitly
  references the canon for its test invocation, so future readers can trace the rule
  from the engine all the way down to the knowledge doc without guessing.

## Why

On 2026-04-28, while validating phase 2 of the Voice subsystem, three concurrent
`dotnet test` invocations against `ZeroCommon.Tests` were started by the harness's
operator (Claude). The first run's output appeared empty, so a second was launched,
then a third. The three orphan testhost processes peaked at 5.3 GB / 3.4 GB / 3.4 GB
RAM and 100% CPU, slowing the user's other work; one run took 21 minutes to finish
because of bin/-directory file-lock contention with its own siblings.

Every run actually passed (72 / 73 / 84 tests, 0 failures), so the lesson is *not*
"tests are flaky" — the lesson is **"how the suite is invoked is part of the
contract."** That belongs in the harness, not in private operator memory.

The fix is two-layered:

1. **Canon** — one knowledge doc that any agent (or human) can read to learn the rule
   and the failure mode. Lives in the knowledge layer (Layer 1) because it's
   methodology, not agent-specific.
2. **Procedure citations** — both agents that invoke `dotnet test` now carry banners
   and step-level citations of specific canon rules, so violations surface at the
   procedure-evaluation step, not after the fact.

A `Test-execution hygiene` axis on each rubric makes the rule auditable in every
agent self-evaluation.

## Migration impact

None. Workflow-only change — no code, build, or DB impact. Existing logs under
`harness/logs/test-sentinel/` and `harness/logs/build-doctor/` are unaffected;
only future runs follow the new procedure.

## Reverse rollout

If the canon proves over-prescriptive (e.g. agents start over-reporting hygiene
failures on harmless cases), revert by:

1. Drop the canon banner + per-step rule citations from both agent procedures.
2. Drop the `Test-execution hygiene` axis from both rubrics.
3. Drop the citation from `release-build-pipeline.md` Step 2.
4. Keep `harness/knowledge/dotnet-test-execution.md` as historical reference; mark
   it `status: archived` in the frontmatter.
5. Bump to v1.1.6 with rollback notes.

Don't delete the canon doc — the incident record is more useful than the rules
themselves if a similar issue recurs.
