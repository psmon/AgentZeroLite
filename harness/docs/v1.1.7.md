**Date**: 2026-04-29

## What changed

### New knowledge document
- **`harness/knowledge/voice-roundtrip-testing.md`** — Methodology
  notes for the pure-model TTS↔STT round-trip test pattern that
  landed in `Project/AgentTest/Voice/TtsSttRoundTripTests.cs`
  (commit `b1a0ec8`). Captures six lessons from the build:

  1. Naive string equality is too strict; raw similarity is too loose.
     Fold case + punctuation + whitespace before comparison so
     tokeniser cosmetics don't mask real content drift.
  2. Whisper auto-corrects semantic typos — the test must report this
     honestly, not paper over it with a higher similarity threshold.
     The maintainer's quick-phrase fixture has `모래` (sand) where
     `모레` (day after tomorrow) is intended; Whisper recognises the
     correct semantic word. Edit distance 1, similarity 96.8%, and
     **the test correctly reports the drift**. That's measurement,
     not flake.
  3. Dual output channel: write to both `ITestOutputHelper` and
     `Console.WriteLine` so the diagnostic trace shows up under any
     test runner / verbosity setting.
  4. Always save WAV evidence on every run. Two artifacts per case
     (`-tts.wav` raw SAPI, `-stt-input-16k.wav` post-resample) make
     failures diagnosable from disk without re-running.
  5. Whisper medium CPU performance baseline recorded for future
     regression detection (1.7–6.3× realtime; xRT improves on
     longer audio because per-call overhead dominates short clips).
  6. Make the resample helper public so the test can call exactly
     what production calls. `WavToPcm.To16kMono` flipped from
     internal to public — no encapsulation secret worth defending.

## Why now

The voice-roundtrip test work surfaced a recurring testing question:
when the model under test is *better than* the test fixture (Whisper
correcting a typo, OCR normalising mojibake, an LLM producing a more
fluent paraphrase), should the test pass or fail? The answer the
knowledge file lands on: **fail with a precise, normalised diff so
the reviewer sees exactly what the model corrected**. A "fuzzy 95%
threshold pass" hides which character changed and why.

This decision needs to be the default the next time someone writes a
similar test, which is why it lives in the agent convention set
rather than in a single test file's comments.
