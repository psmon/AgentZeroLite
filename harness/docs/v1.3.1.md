# v1.3.1 — Mission log directory renamed to English

**Released**: 2026-05-02
**Bump rationale**: Patch — directory rename + documentation tightening.
No behavioral change to dispatch, triggers, or completion-log contract.

## What changed

| Before (v1.3.0) | After (v1.3.1) |
|---|---|
| `harness/logs/미션기록/` | `harness/logs/mission-records/` |
| `missions_log_dir` config field pointed to Korean path | now points to English path |

Reasoning (operator feedback):

> "디렉토리명은 영문... 하위 생성파일을 언어에 대응하면됨"

The directory carries paths through tools that don't always handle mixed-script
gracefully (some shells, CI logs, cloud editors, third-party indexers). Since
the *file slugs inside* still match the mission's language, the operator's
language preference is preserved at the file level — only the container name
gets the English-default treatment that every other harness path already has.

## File slug policy — clarified

The protocol now explicitly distinguishes between **directory names**
(English, tooling-safe) and **file slugs inside** (operator's language OK):

| Path piece | Language |
|---|---|
| `harness/missions/` (dir) | English |
| `harness/missions/{filename}` slug | English (`M0001-add-missions-system.md`) |
| `harness/logs/mission-records/` (dir) | English |
| `harness/logs/mission-records/{filename}` slug | **Mission's language OK** (`M0001-수행결과.md` for ko, `M0002-execution-result.md` for en) |
| File body content | Per-mission language matching (unchanged) |

The asymmetry is deliberate: mission filenames are external indices (referenced
by operator triggers like "M0042 수행해" — slug rarely surfaces), while
completion-log filenames live alongside the body content the operator reads,
and language continuity at the filename helps when scrolling a directory listing.
