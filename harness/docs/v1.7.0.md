# v1.7.0 — OS-control surface (mission M0014)

**Released**: 2026-05-07
**Bump rationale**: Minor — new engine `os-cli-e2e-smoke`, new
`_shared/os-control.md` knowledge entry, expanded LLM tool catalog
(six new `os_*` tools), expanded CLI verb table (one new `os` group
with 11 verbs). No agent added, no trigger removed; existing flows
unchanged.

## What's new

### `harness/engine/os-cli-e2e-smoke.md` (new)

End-to-end probe that drives `AgentZeroLite.exe -cli os <verb>` to
verify a fresh build is reachable from the desktop. Read-only —
`기능접근만 확인` per the mission contract. Steps: `list-windows`
→ `get-window-info` → `screenshot` → `element-tree` → `dpi`. Artefacts
land under `tmp/os-cli/`. Triggers: `OS CLI 스모크`, `os smoke`,
`launch-self-smoke`, `M0014 스모크`.

Evaluation rubric covers reachability, capture integrity, UIA tree
reachability, audit-trail intact, and "no input-simulation drift" —
the smoke must NOT touch SendInput.

### `harness/knowledge/_shared/os-control.md` (new)

Single source of truth for the OS-control surface — CLI verb table,
LLM tool table, approval gate semantics, audit-log schema,
artefact paths under `tmp/os-cli/`, screenshot encoding policy,
anti-patterns. Cross-referenced from `Docs/OsControl.md` (public)
and `Docs/agent-origin/03-adoption-recommendations.md` §P1-2.

### `tmp/os-cli/` structure

```
tmp/os-cli/
├── audit/<date>.jsonl       every CLI/LLM call (caller, verb, args, ok)
├── screenshots/<date>/      PNG outputs
├── e2e/<date>.log           E2E smoke summary
└── (scripts live at Docs/scripts/launch-self-smoke.ps1 — tracked in git)
```

The repo root is detected by walking up from `AppContext.BaseDirectory`
looking for `.git/` or `AgentZeroLite.sln`. Installed-path fallback:
`%LOCALAPPDATA%\AgentZeroLite\tmp\os-cli\`.

### Approval gate (input simulation only)

Two enabler signals, either is sufficient:
1. CLI: `--allow-input` flag per call
2. Process env: `AGENTZERO_OS_INPUT_ALLOWED=1`

Read-only verbs (window enum, screenshot, UIA tree, DPI, activate)
are unconditional. There is no per-call human prompt — the audit log
is the forensic record.

### LLM-side anti-recursion

`os_launch` (i.e. starting the app from inside an LLM session) was
deliberately NOT added to the toolbelt. The CLI keeps `open-win` for
operator use; exposing it to the model would create a self-spawn
loop. Future missions wanting this need their own gate.

## Why a Minor (not Patch)

Per harness convention:
- **Patch**: hierarchy-internal additions, doc tightening
- **Minor**: new agent OR new engine OR new tool surface
- **Major**: architecture change

This adds **one new engine** + **one new knowledge file** + **a
substantial LLM tool surface (6 tools, ~25% expansion of KnownTools)**.
That clears the Minor bar on every count. Major was considered for
the input-simulation gate (a new security primitive), but the gate is
local to the OS surface — no other harness mechanism depends on it,
so the bump stays Minor.

## Compatibility

No breaking changes. The `IAgentToolbelt` interface gained six methods
with default implementations that return a `not supported` JSON
envelope, so `MockAgentToolbelt` (and any future toolbelt) keeps
compiling untouched. KnownTools / GBNF additions are additive — the
existing 5-tool surface continues to parse.

The completion log for M0014 lives at
`harness/logs/mission-records/M0014-수행결과.md`.
