## 1. Injection surface (text → ConPTY)

Reviewed: `Project/AgentZeroWpf/UI/APP/MainWindow.xaml.cs:1855-1876` (ExePath quoting fix from 365ffec).

The diff:
- Trims wrapping `"…"` from `tab.ExePath` (`exePath = exePath[1..^1]`).
- Wraps in quotes only when whitespace is present (`IndexOfAny([' ', '\t']) >= 0`).
- Emits `cmd /c "set "PATH=…"&&pushd "{workDir}"&&{quotedExe} {Arguments}"`.

Findings:

- **F-1 [Info / CWE-78] Argument concatenation in ConPTY launch is unescaped — pre-existing baseline.** `tab.Arguments` is interpolated raw at line 1859, and the new ExePath quoting only escapes whitespace, not embedded `"`/`&`/`|`/`>` metacharacters. A user-edited `CliDefinition` with metacharacters in Arguments or ExePath would break out of the `cmd /c` string and run extra commands. **Confirmed baseline** — `git show v0.3.0:…MainWindow.xaml.cs` shows the same `$"{tab.ExePath} {tab.Arguments}"` raw concat. Threat is local-user-self-injection only (editing CliDefinition requires opening the editor and typing). Per `harness/knowledge/security-surface.md`: "Physical access to the host — out of threat model." Diff *improves* the path-with-space case without making metachars worse. **Not release-blocking.** Future hardening: route through `ProcessStartInfo.ArgumentList` or build a real cmd-quoter (`CmdEscape(arg)`).
- **No new `Process.Start` / `WriteToTerm` / write-channel call sites added** — diff does not expand the injection surface (`grep '^\+.*Process\.Start|^\+.*WriteToTerm' = 0` matches).
- **AgentBot keystroke broker unchanged** — no diff in `AgentBotActor`/`AgentBotWindow`/`AgentReactorActor`. Existing CHT/KEY-mode paths preserved.

## Additional review — harness-view JS (M0001 / M0002 missions feature)

Files: `Home/harness-view/scripts/build-indexes.js` (+111), `Home/harness-view/js/views/missions.js` (+359).

- **`parseSimpleFrontmatter`** parses YAML-ish frontmatter into a flat string map. Skips blank lines, comments, indented continuations. Strips wrapping quotes. **No code execution path** — values are stored in JSON and rendered later via `h(tag, …, textNode)` with `createTextNode` (see `Home/harness-view/js/utils/dom.js:18`). Mission frontmatter values cannot reach `innerHTML`.
- **`missions.js` DOM rendering** — only two `html:` (innerHTML) sites in the entire view: line 119 uses `ICONS[meta.icon] || ICONS.circle` (lookup from a static map; `meta.icon` is one of six known keys from `STATUS_META`, not user-controlled — frontmatter `status` only feeds `statusMeta`'s key lookup); line 320 uses the constant `ICONS.file`. **No XSS surface from mission frontmatter.**
- **Path traversal** in `loadMd(it.file)`: `it.file` is constructed by the build-side as `${PATHS.missions}/${f}` where `f` comes from `fs.readdirSync(missionsDir).filter(f => f.endsWith('.md'))`. Filenames originate in the trusted repo working tree; the URL hash (`#missions/{id}`) does not feed file-path construction (it's matched against `index.items` by strict equality on `it.id`). **No traversal exposure.**
- **F-2 [Info / CWE-79] Static viewer renders MD bodies via `marked.parse → innerHTML` without sanitization — pre-existing baseline.** `Home/harness-view/js/components/md-viewer.js:99` does `h('div', { class: 'md-viewer', html: renderMarkdown(state.content) })`. The Missions feature reuses this component, expanding the corpus to `harness/missions/` + `harness/logs/mission-records/`. Threat: a maintainer-merged malicious MD file could XSS the static viewer. **Same threat already exists for** `harness/logs/`, `Docs/`, `harness/agents/`, etc. since long before v0.3.0. **Not introduced by this release; not blocking.** Future hardening: add DOMPurify to the viewer pipeline. Tracking suggested as a knowledge note, not a release blocker.
