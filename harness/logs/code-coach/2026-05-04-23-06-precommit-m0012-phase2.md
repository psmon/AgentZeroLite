## Failure-mode review

- **Concurrent collector access** — `TokenRemainingCollector` and `SessionHeartbeatCollector` both scan
  `cc-hud-snapshots/**/*.json` recursively. SQLite WAL handles concurrent readers/writers; both write
  to **different tables** so no row contention. Stagger (5s vs 7s) plus 30s/15s cadence keeps overlap rare.
  ✓ acceptable.
- **`EnsureWrapperOnDisk` always overwrites** — small file, idempotent content for same build.
  No race with running wrappers (file replaced atomically by .NET; running wrapper has the script in
  memory).
  ✓ acceptable.
- **Heartbeat prune** — `WHERE LastSeenUtc < UtcNow - 1day` runs every 15s tick. Cheap on the
  `(LastSeenUtc DESC)` index. ✓ acceptable.
- **Wrapper v3 first-tick cleanup** — deletes `cc-hud-snapshots/{account}.json` flat file. If a v2
  wrapper is concurrently writing it (during version transition), the delete might race the write —
  but worst case the v2 write recreates it and v3 deletes again next tick. Eventually consistent.
  ✓ acceptable.
- **WrapperScript SessionId validation** — strips non-`[A-Za-z0-9\-_]` and caps at 64 chars. Claude
  Code session_id is a UUID (32 hex + 4 dashes = 36 chars), well within bounds. ✓ tight.
- **WebDevBridge bounds** — `windowMinutes` clamped to `[1, 60]` server-side and matched in zero-bridge.
  ✓ defensive on both ends.
- **navigator.clipboard.writeText** — async, may throw on permission denial (some WebView2 configs).
  Wrapped in try/catch, banner on failure. ✓ degrades gracefully.

## Findings

### Suggestion 1 — `as--qa` heuristic class is mockup-leakage

**File**: `Project/Plugins/token-remaining/widget.js:336` (renderActivePanel)

```js
const acctCls = (s.accountKey || '').toLowerCase().includes('qa') ? ' as--qa' : '';
```

The `as--qa` class swaps the account-pill color to a darker orange, so the operator can visually
distinguish a "qa-only" account from a "personal" one. But the trigger is a substring match on
"qa" — false-positive on `claude-aqa`, `claude-quartz`, etc. The .pen mockup used a literal
`claude-qa` row to demo the second color; the JS port baked that example into a heuristic.

**Suggested fix**: drop the heuristic for now (mockup-only convention). Either remove the class
entirely or replace with a stable signal (e.g., user-tagged accounts in Settings). One-line removal:

```js
const acctCls = '';   // future: read from prefs.accountColors[s.accountKey] if user-tagged
```

Severity: **Suggestion** — cosmetic only, no functional impact; defer-able to a follow-up.

### Suggestion 2 — Comment in App.xaml.cs overstates `EnsureWrapperOnDisk` behavior

**File**: `Project/AgentZeroWpf/App.xaml.cs:117-121`

```csharp
// M0012: keep wrapper.js on disk in sync with this build's
// embedded WrapperScript (sha256 compare → rewrite-on-diff).
```

The comment claims sha256-compare-then-rewrite, but `EnsureWrapperOnDisk()` (`StatusLineWrapperInstaller.cs:347-353`)
always calls `File.WriteAllText`. There's no hash short-circuit. Behavior is correct (idempotent on
identical content) but the comment misleads a future reader.

**Suggested fix** — pick one of:
1. Drop the parenthetical: `// embedded WrapperScript (idempotent overwrite per app launch).`
2. Implement the short-circuit in `EnsureWrapperOnDisk`:
   ```csharp
   var current = File.Exists(WrapperPath) ? File.ReadAllText(WrapperPath) : null;
   if (current == WrapperScript) return;
   File.WriteAllText(WrapperPath, WrapperScript);
   ```

Severity: **Suggestion** — comment-fix; option 1 is the smaller change.
