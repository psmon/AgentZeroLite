# M0024 수행결과 — Phase 1 (scaffolding) + Phase 4 (harness 전문가 배치)

## 실행 요약

미션 brief 가 요구한 capability = **음성에 화자(speaker) 라벨을 붙여서 음성노트
+ Voice 테스트에서 누가 무슨 말 했는지 보이게 하기**. 모델 후보는 brief 에서
microsoft/VibeVoice 가 명시됐고 "Music 모델 도입한 것처럼 embed, pip 없이" 가
강조됨.

조사 결과 두 요구사항이 충돌함을 발견 — VibeVoice-ASR (화자 구분하는 변종) 은
7B params + 공식/커뮤니티 모두 ONNX export 없음. AgentZero 의 86M AST 패턴
("ONNX 1회 다운로드, ZeroCommon embed") 에 들어갈 수 없는 스케일.

operator 와 AskUserQuestion 으로 **Sherpa-ONNX (pyannote-segmentation-3-0 + 3D-Speaker)** 로 노선 변경 합의. 이 모델 조합은:

- 같은 capability (DER ~4% 수준 실용 정확도) 달성
- 총 ~50 MB — AST 보다도 가벼움
- 공식 C# NuGet (`org.k2fsa.sherpa.onnx`) 보유
- "pip 없이 embed" 요구 완벽 충족

본 라운드는 **Phase 1 (코드 scaffolding)** + **Phase 4 (전문가 + 지식)** 만
수행. Phase 2 (Voice 탭 UI) 와 Phase 3 (voice-note 플러그인 통합) 은 다음
라운드 — operator 가 seam 모양을 검토한 후 진행하는 게 안전.

### Loopback 동작의 특이점

System audio 는 continuous 라서 VAD 가 의미 없음. Loopback 소스를 선택하면
무조건 **manual** 모드처럼 거동 — START → 시스템 출력 계속 buffer → STOP 누르면
buffer 전체를 한 번에 STT + diarize → transcript 한 줄 출력. Auto VAD 라디오
버튼 선택 여부와 무관. 이 거동은 `OnVoiceTestStartStop` 의 `loopbackActive`
분기 + status 라인 ("Capturing system output (WASAPI loopback) — press STOP
to transcribe.") 으로 표면화.

### 동작

캡처 시작 → 10초 timer → tick 마다:
1. `_pipelineBusy` 면 skip (main pipeline 과 충돌 방지)
2. `_partialBusy` interlocked CAS → 동시 1개 보장
3. 활성 capture (`_voiceLoopback` 우선, 없으면 `_voiceCapture`) 에서 `PeekPcmBuffer()`
4. 길이 < 2 s → skip
5. 길이가 직전 tick 과 같음 → skip (새 audio 없음)
6. `Task.Run` 으로 STT (캐시된 `_partialStt` 재사용 — Whisper 매 tick 재워밍 방지)
7. 결과를 transcript 에 `  … (partial @ Ns) <text>` 형식으로 append

STOP / utterance 종료 시:
- Timer stop
- Main pipeline 은 그대로 ConsumePcmBuffer (drain) 후 full STT + diarize → `You: [Speaker A] <text>` 출력
- Partial 라인은 transcript 에 남아 있음 (rolling preview 의 역사)

### 동작 흐름

```

        │ utterance PCM
        ▼

        │ majority speaker
        ▼

        ▼
```
