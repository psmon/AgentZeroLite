# M0029 수행결과 — 에이전트 악단 로컬 MP3 플레이리스트

## 실행 요약

미션 brief = 유튜브 외 **로컬 MP3** 소스 추가: 폴더 스캔 → mp3태그+파일명+폴더명 기반
LLM 분류 → 유튜브와 분리된 **SQLite 영속 재생목록** → 상단 소스 탭으로 전환,
하단 시각화 밴드는 동일 작동. MP3는 비전이 없으므로 **일반 플레이어 모드(악기·가수) 전용**.

- **확장 1**: 재생 중 환경음(악기)을 AST가 파악 → 트랙 **보유악기를 실시간 누적 영속**,
  목록에서 **악기명 검색 필터** ("피아노 포함" / "피아노만 있음(단독)").
- **확장 2**: 스캔은 **비동기 백그라운드 잡** — 진행 이벤트 스트림으로
  **스캔 중에도 목록이 증분 갱신**되어 스캔된 곡부터 즉시 재생 가능.

## 결과

### 아키텍처 결정 3가지 (기존 구조 재사용이 핵심)

1. **오디오 배선 무변경** — 밴드는 SystemLoopback(OS 믹서) 캡처로 구동되므로
   `<audio>` 재생만으로 유튜브와 동일하게 반응. M0026과 같은 원리.
2. **재생은 가상 호스트** — 브릿지가 스캔 루트를 `https://mp3.local/`로
   `SetVirtualHostNameToFolderMapping` → 오디오 바이트가 postMessage 브릿지를 타지 않고
   네이티브 스트리밍(시킹 포함). 폴더 변경 시 재매핑.
3. **분류는 M0026 패턴 재사용** — `LlmGateway.OpenSession()` 1회용 세션 +
   화이트리스트 클램프(`MatchCategory`). 힌트 = 태그(제목/아티스트/앨범/장르) +
   파일명 + 폴더명 (`Mp3Scanner.BuildClassifyHint`). LLM 꺼짐 → 태그장르 클램프("tag") →
   마지막 카테고리("fallback") 순 폴백, 출처 배지(✦/🏷/≈)로 가시화.

## 후속 수정 #1 — MP3 재생 실패 (스트리밍 전환)

**운영자 보고 (원문)**: "스캔은 되는데 재생은 실패됨.. 스캔처리 된 녀석중에서 확인해
재생 기능 확인해볼래? 스트리밍 mp3 재생이 필요할지도"

**진단**: `SetVirtualHostNameToFolderMapping`은 Chromium **미디어 스택의 HTTP Range
요청(부분 요청)에 응답하지 못하는** WebView2의 알려진 제약이 있다. `<audio>`는 재생 시
`Range: bytes=…`를 보내므로 미디어 로드만 실패 — 일반 파일 fetch(스캔·목록)는 정상이라
증상이 정확히 "스캔 OK / 재생 실패"로 갈린다. 운영자 추측(스트리밍 필요)이 맞음.

**수정** (플러그인 v0.16.1):
- `WebDevBridge.cs` — `https://mp3.local/*`에 `WebResourceRequested` 인터셉터 추가.
  파일을 `FileStream`으로 직접 서빙하며 **HTTP Range 완전 지원**: `bytes=s-e` /
  `bytes=s-` / `bytes=-suffix` → `206 Partial Content` + `Content-Range`
  (`StreamSlice` 래퍼로 요청 구간만 무복사 스트리밍), Range 없으면 200 전체,
  잘못된 구간은 416. 경로는 unescape 후 스캔 루트 하위로 클램프(`..` 차단),
  `*.mp3`만 서빙. 폴더 매핑은 구버전 런타임 폴백으로 유지.
- `agent-band.js` — `<audio>` error 이벤트를 코드명 배지로 표시
  (network/decode/src-not-supported) — 조용한 재생 실패 제거.

**빌드 재검증**: 호스트 빌드 ✅ (오류 0 · 경고 0), `node --check` ✅,
로컬 설치 zip(v0.16.1) 재생성 ✅.

## 후속 수정 #2 — 2단계 스캔(LLM 비동기 분리) + 미니 플레이어 UX

**운영자 보고 (원문)**: "스캔은 LLM없이 빠르게 먼저 진행하고..미분류로 먼저등록....
LLM을 비동기로 미분류인항목에서 뽑아 분석업데이트하면 더 좋지 않을까? 분석이 완료될때까지
음악리스트를 못보는 불편함을 해소....mp3의 경우 플레이리스트가 더 많은 ux가 필요하니
조정도... mp3파일정보를 보여주는 창이 너무 큼 파일정보 창은 미니멀하게(play,pause,메타정보,
커버이미지있으면그것도)...플레이기이고 우축 플레이 리스트는 조금더 크기 확장해죠"

**진단**: v0.16의 스캔 잡이 파일마다 태그 읽기 직후 LLM 분류를 **인라인 대기**하는 구조라,
느린 로컬 LLM에서는 곡당 수 초씩 목록 채움이 지연됨 (재생은 가능했지만 전체 목록 파악 불가).
UX는 유튜브 페인 크기를 그대로 물려받아 플레이어가 과대.

**수정** (플러그인 v0.17.0):
- `WebDevHost.Mp3.cs` — 스캔 잡을 **2단계 분리**: 1차 패스 = LLM 없이 태그만 빠르게
  전곡 upsert(미분류, `CategoryBy="none"`) + 즉시 `mp3.track` 발행 → 2차 패스 = 미분류
  백로그를 순차 LLM 분류, 판정마다 `mp3.track` 라이브 갱신. 진행 이벤트 `phase`
  ("scan" → "classify") + 단계별 done/total. 취소 시 잔여 미분류는 다음 스캔에서 재시도.
- 커버 아트 — `Mp3Scanner.ReadCover`(ID3 APIC) + `TryGetMp3Cover(id)` +
  브릿지 `https://mp3.local/__cover/{id}` 라우트 (id 기반 DB 조회, 경로 미개입).
- `agent-band` UI — 플레이어를 **미니 카드(고정 280px)**로 축소: 커버(APIC, 💿 폴백) +
  제목/아티스트/분류 배지/악기 칩 + `<audio>` 컨트롤만. `#yt-top.mp3-mode`에서
  **플레이리스트가 나머지 폭 전부** 차지. 상태 배지 단계 표시(📂 스캔 → 🧠 분류),
  **'미분류' 탭** 추가(2차 패스 대기 곡 모아보기).

**빌드 재검증**: 호스트 빌드 ✅ (0/0), 헤드리스 298 pass ✅, `node --check` ✅,
로컬 설치 zip(v0.17.0) 재생성 ✅.
