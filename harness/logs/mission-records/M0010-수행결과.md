# 수행 요약

WebDev 페이지 안에 갇혀 있던 sample WebView2 를 새 창(`FloatingWebDevWindow`)
으로 분리할 수 있게 만들었음. 분리는 reparent 방식 — M0006 의 ViewSlot 캐시
구조를 깨지 않고 동일 WebView2 인스턴스를 새 창의 Body Grid 로 옮기기만 하므로
JS 상태 / 오디오 / 스크롤이 그대로 유지된다. 창을 닫으면 자동 도킹 복귀.

새 창 자체는 우클릭 컨텍스트 메뉴로 다음을 토글 — Pin (Always-on-top) /
Transparency (50–100% 슬라이더) / Title bar / Reload / DevTools / Dock back.
설정 + 창 위치/크기는 sample id 별로 `%LOCALAPPDATA%/AgentZeroLite/WebDev/
floating-prefs.json` 에 영구 저장되어 같은 sample 을 다시 Detach 할 때
마지막 상태로 복원된다 (현재 작업영역 안인지 검증한 뒤 clamp).

Title bar OFF 상태의 이동은 펜슬 디자인 협의대로 **Alt + 좌클릭 드래그** 로
한정. Win32 `GetAsyncKeyState(VK_MENU)` 폴링(60 ms) 으로 Alt 가 눌릴 때만
WebView2 위에 투명 오버레이를 띄워 클릭을 가로채고 `DragMove()` 한다 — 일반
좌클릭은 정상적으로 WebView2 로 전달되어 웹 콘텐츠 조작에 영향 없음.

리사이즈는 chrome 상태와 무관하게 `WM_NCHITTEST` 후크로 6px 가장자리 8방향
hit-test 를 항상 활성화.

### B. 새 윈도우 (`Project/AgentZeroWpf/UI/Components/FloatingWebDevWindow.xaml(.cs)`)

- **창 chrome**: `WindowStyle="None"` + `AllowsTransparency="True"` +
  외곽 둥근 Border + 드롭섀도. titlebar / footer 는 chrome 토글로
  `RowDefinition.Height` 0/32/26 만 바꿔 표시·숨김 (창 자체는 재생성 X).
  Resize 가능, MinWidth/Height = 320/220.
- **커스텀 titlebar** (chrome ON): 좌측 상태점 + "WebDev — {sample}",
  우측 Pin / Dock back / ✕ (✕ 도 dock back alias).
- **우클릭 ContextMenu**: Pin / Title bar 토글 / Transparency 서브메뉴
  (220px Slider, 50–100%, snap 0.05) / Reload / DevTools / Dock back.
- **Drag 처리**:
  - chrome ON → titlebar `MouseLeftButtonDown` → `DragMove()`.
  - chrome OFF → DispatcherTimer 60 ms 가 `GetAsyncKeyState(VK_MENU)`
    체크 → Alt 눌림 시 `dragOverlay` (Border, 반투명 #0A84FF22 + 점선
    안내 텍스트) Visibility = Visible → 좌클릭 캡처해서 `DragMove()`.
    Alt 해제되면 즉시 hide → WebView2 가 다시 클릭 받음.
- **Resize**: `WM_NCHITTEST` 후크에서 6px 가장자리 8방향 hit-test.
  `PointFromScreen` 변환 + try/catch (visual disconnect 보호).
- **Persist 트리거**: `OnLocationChanged` / `OnRenderSizeChanged` /
  Pin/Opacity/Chrome 변경 / `OnClosed` — 매번 `WebDevFloatingPrefs.Save`.
- **Dock back**: `Close()` → `OnClosed` 에서 `webHost.Children.Remove(_view)`
  → `_onDockBack(sampleId)` 콜백. 콜백 안에서 host (WebDevPagePanel) 가
  WebView2 를 다시 viewHost 로 옮긴다. WebView2 는 절대 dispose 안 함.
- **Restore on Detach**: `WebDevFloatingPrefs.Get(id)` 로 복원, 좌표는
  `IsRectOnAnyScreen` 으로 현재 모니터들과 64-px 헤더 sliver 가 교차할
  때만 복원 (모니터 빠진 경우 fallback = owner + 80,80).

- **`OnUninstallClick` 보강** — 플러그인 uninstall 시점에 그 sample 이

  Directory.Delete 가 돌아감).
- **breadcrumb** — 활성 sample 이 floating 일 때 "WebDev › sample · floating"

## 후속 수정 #1 (operator 1차 검수 — Detach 후 미아 + 새 창 invisible)

operator 1차 보고: **Detach 누르면 새 창을 찾을 수 없음. Detach 후 토글 버튼이
유지되어 sample 이 미아가 됨**.

### 진단

1. **새 창 invisible** — `AllowsTransparency="True"` + `WindowStyle="None"` 으로
   layered window 가 되면 WebView2 (HwndHost 기반) 가 렌더 안 됨. WPF 의 잘
   알려진 airspace 제약 — layered window 안에서는 native HWND 자식이 표시 X.
   창은 떠 있었지만 WebView2 가 안 보여서 "창이 없는 것처럼" 보였음.
2. **토글 버튼 미작동** — 1번 때문에 `floating.Show()` 후 다음 라인이
   실행되긴 했지만, 사용자가 "보이는 새 창" 을 못 찾으니 placeholder 가
   떠 있어도 어디에서 dock back 해야 할지 헷갈림. 토글 어포던스가
   placeholder 안에만 있어서 토글바에는 여전히 "Detach" 가 노출.

### 수정

```
System.NullReferenceException at FloatingWebDevWindow.OnOpacityChanged
  at System.Windows.Controls.Slider.OnValueChanged
  at ...OnMinimumChanged
  at FloatingWebDevWindow.InitializeComponent()
```

XAML 의 `<Slider Minimum="0.5" Value="1.0">` 파싱 중 — Minimum 이 먼저 적용되면
Slider 가 default `Value=0.0` 을 0.5 로 coerce 하면서 `ValueChanged` 가 동기 fire.
그 시점은 `InitializeComponent()` 한가운데이고, 생성자의 `_pref =
WebDevFloatingPrefs.Get(...)` 줄은 아직 실행 안 됨 → `_pref.Opacity = e.NewValue`
에서 NRE.

### 수정

- XAML 의 `<Slider ... ValueChanged="OnOpacityChanged">` 속성 제거
- 코드: 생성자의 `_pref` 초기화 + `ApplyPrefBeforeShow()` 끝난 뒤
  `sldOpacity.ValueChanged += OnOpacityChanged;` 로 wiring
- 진단 보강 — `OnDetachClick` catch 블록이 풀 stack trace 를 클립보드에 자동 복사
  + MessageBox 안내문에 "붙여넣어 달라" 명시 → 다음 회귀 시 1-shot 진단

### 빌드 재검증

- `dotnet build … -c Debug` — **0 errors**.

## 후속 수정 #3 (operator 3차 검수 — chrome OFF stuck + DevTools 팝업)

operator 3차 보고: **새 창 모드 타이틀바 없애기 하면 titlebar + footer 가 모두
사라져 우클릭 진입경로가 없음. 컨텐츠 영역 우클릭하면 WebView2 의 DevTools 팝업이
뜸. 8-px 파란 strip 도 안 보임 (어디에 있는지 모르겠음)**.

### 진단

1. **WebView2 default 컨텍스트 메뉴 가 우리 메뉴 가림** — 기본 활성 상태라 우클릭
   = 브라우저 메뉴 (Save, Inspect…) 표시.
2. **chromeHandle (8-px strip) 이 안 보임** — `Grid.RowSpan` + `VerticalAlignment="Top"`
   으로 webHost 위에 z-order 로 올렸지만 WebView2 (HwndHost) 의 native HWND
   airspace 가 또 가려버림. 이전 미션의 같은 함정.
3. **dragOverlay 기반 Alt 진입경로** 는 60ms 폴링이라 빠른 Alt+우클릭 시점에
   overlay 가 안 떠 있어서 클릭이 WebView2 로 흘러가버림 — 사실상 미작동.

### 수정

- WebView2 기본 컨텍스트 메뉴 OFF — `view.CoreWebView2.Settings.AreDefaultContextMenusEnabled = false`
  (WebDevPagePanel 의 view 생성 시점, try/catch 로 구버전 SDK 안전).
- chromeHandle 을 **자기 row** (rowHandle, 0/8px 토글) 로 분리 — webHost 와 물리적
  으로 겹치지 않게 해서 HwndHost airspace 에 안 가려짐.
- WebView2 의 `CoreWebView2.ContextMenuRequested` 이벤트로 직접 후킹 — 우클릭
  즉시 fire, Alt 상태 체크 후 `args.Handled=true` + Dispatcher.BeginInvoke 로
  ContextMenu 띄움. 폴링 race 제거.

## 후속 수정 #4 (operator 4차 검수 — Alt 제거 + 작은 titlebar + Pin 독립)

operator 4차 보고: **상당 작은 titlebar 항상 노출이 좋겠음. Alt+우클릭 / Alt+드래그
는 event-bubble 사이드이펙 우려 + 실제 미작동이라 모두 제거. Pin 모드 상태에서
메인 윈도우 minimize 하면 floating 도 같이 내려감 → Pin 일 때는 메인 minimize 영향
받지 않게 유지**.

### 수정

- titlebar 를 **24-px 컴팩트** 로 축소 (이전 32-px), Pin/⤡/✕ 버튼도 단일 글리프
  로 슬림화. WindowChrome.CaptionHeight 도 24 로 매칭.
- footer 도 26 → 20-px 로 컴팩트화.
- Alt 관련 코드 **전부 제거**:
  - `dragOverlay` Border + `OnDragOverlayDown/RightDown` 핸들러
  - `DispatcherTimer` + `GetAsyncKeyState(VK_MENU)` 폴링
  - `OnViewContextMenuRequested` (WebView2 후킹)
- WebView2 `AreDefaultContextMenusEnabled = false` 는 유지 (DevTools 팝업 차단
  자체는 유의미).
- **Pin = Owner detach** 패턴 도입 — `OnPinToggle` 에서 `ApplyOwnerForPinState()`
  호출:
  ```csharp
  Owner = _pref.Topmost ? null : _initialOwner;
  ```
  Pin ON → `Owner=null` → 메인 minimize/restore/activation 영향 0. Pin OFF →
  Owner 재부착, 메인 닫히면 같이 닫힘. 생성자에서도 prefs 의 Topmost 값에 따라
  초기 Owner 결정 (Pin 상태 복원이면 owner-less 시작).
- `_initialOwner` 필드로 owner 캐시.

### 빌드 재검증

- `dotnet build … -c Debug` — **0 errors**.
