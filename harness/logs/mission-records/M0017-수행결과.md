# 수행 요약

AgentBot 의 AI 모드에 **위임 모드(Delegation Mode)** 를 추가했습니다. 음성으로
"에이전트 위임" / "에이전트 위임중단" 을 말하면 상태가 토글되어, 위임 모드일 때는
사용자의 모든 발화가 `Claude` 라는 이름의 터미널 탭으로 강제로 전달됩니다. Claude
의 응답은 AgentLoop 의 `done` 메시지로 한국어 요약되어 기존 TTS 경로(`AddBotMessage`
→ `SpeakBotMessageIfEnabled`)를 그대로 타고 음성으로 출력됩니다.

기존 AgentToolGrammar 의 **Mode 2 (Terminal Relay)** 가 이미 send_to_terminal →
wait → read_terminal → done 사이클을 구현하고 있어서, 새 도구나 새 펑션콜을
추가하지 않고 **AgentBotActor 에서 StartAgentLoop 메시지를 래핑** 하는 식으로
액터 상태머신을 그대로 살렸습니다. 따라서:
- 위임 모드 ON 일 때 → `AgentBotActor.WrapForDelegation(userRequest)` 가 강한
  Mode 2 지시문을 사용자 발화 앞에 붙여 자식 `AgentLoopActor` 로 위임.
- 위임 모드 OFF 일 때 → 평소처럼 `StartAgentLoop` 가 그대로 통과 (Mode 1/2 혼합).

## 비고

### 사전조건 체크 — 1차 범위
미션 본문의 "playwright 설치 여부 확인 → Claude 에게 설치 요청 → 설치 중 허용 프롬프트에 엔터 전송" 시나리오는 1차에서는 **구현 보류** 했습니다. 이유:

1. 위임 모드의 **본질은 라우팅** 입니다. playwright 설치는 부속 능력으로, 라우팅이 동작한 다음에야 의미가 있습니다. 핵심 가치(라우팅 + 음성 입출력 폐쇄 루프)를 먼저 합격선까지 올린 뒤 후속 라운드에서 부속 능력을 추가하는 것이 안전합니다.
2. 설치 중 발생하는 Claude CLI 의 허용 프롬프트는 ANSI/스파이너로 표현되어 `ApprovalParser` 만으로는 견고하지 않습니다 — 별도의 패턴/엔터 자동화가 필요합니다.
3. Claude CLI 인증 자체는 미션에서 PASS 로 명시 → 인증 흐름은 무시.

대신 1차에서는 **터미널 사전 점검**(`FindClaudeTerminalLabel`) 만 구현하고, 없을 때 음성으로 안내합니다. playwright 능력 확인 / 설치 자동화는 후속 미션 후보로 남깁니다.

### 사용 흐름 (시각장애인 사용자 관점)

1. 마이크 ON (`btnVoiceMic`) → AI 모드 진입 후 "에이전트 위임" 발화.
2. AgentBot: "위임 모드를 시작합니다. 이후 음성 요청은 Main / Claude 으로 전달됩니다." (음성)
3. "쿠팡에서 40대가 좋아하는 물건 찾아줘" 발화.
4. AgentLoop → list_terminals → send_to_terminal(Claude) → wait → read_terminal → done.
5. AgentBot: "Claude 가 응답한 결과 요약" (음성, 1~2 문장).
6. "에이전트 위임중단" 발화 → 평소 AI 모드로 복귀.

### 인텐트 분류 우선순위

`VoiceCommandInterceptor.Classify` 순서: **Stop → Delegation → Summarize → PassThrough**.
"그만" 단독은 여전히 TTS 인터럽트로 잡힙니다 (Detects_stop 테스트 + `Stop_alone_is_still_stop_not_delegation`).

## 후속 수정 #9 — 위임 모드에서 LLM 우회, 음성을 Claude 에 직접 입력 (2026-05-12)

### Operator 요청

> 위임모드일때 AI가 별도 해석하는게 아닌.. DONE만 그대로 읽어주는게 좋을것같음 로그
> 파악하면 DONE을 그대로 TTS하면되고 AIMODE는 별도의 처리를 안해도될것같음 AIMODE의
> 일반대응 펑션콜과 겹치면서 오히려 정보요약이 안됨

위임 모드에서 사용자 발화는 LLM 의 wrap 지시문 → list_terminals → send_to_terminal → wait →
read_terminal → done 루프를 거쳤음 (후속 #1·#2·#5). 이 흐름이:
- 5~10초 추가 latency (외부 백엔드 한 사이클)
- wrap 지시문이 모델의 평소 tool-calling 본능과 충돌 (모델이 wrap 을 무시하고 Mode 1
  으로 답하거나, send_to_terminal 자리에 다른 tool 을 호출)
- Claude 가 이미 `DONE(...)` 으로 깔끔하게 응답 wrapping 해주는데 거기서 또 LLM 한 사이클
  돌려서 "요약" 하려고 함 — 결과적으로 요약이 더 부정확

후속 #5 의 DONE → TTS 단축경로는 잘 동작. 송신 쪽도 LLM 을 우회하고 Claude 세션에 직접
쓰면 round-trip 이 정확해짐.

### 새 흐름

```

[Voice 분류기]  PassThrough (Stop/Delegate/Summarize 아님)
[Input]    SendCurrentInput → ChatMode.Ai 분기
            ├─ _delegationMode == true?
            │    └─ 예: TryRelayDirectlyToClaude(text)
            │         ├─ FindClaudeTerminal → (g=1, t=0, "Claude")
            │         ├─ ConPtyTerminalSession.WriteAndSubmit(text)   ← LLM 우회
            │         ├─ Tell MarkTerminalIntroduced + MarkConversationActive
            │         └─ UI: 사용자 bubble + "→ Claude (delegation direct)" 브레드크럼

[Claude]   ... 답변 준비 ...
           AgentZeroLite.exe -cli bot-chat "DONE(오늘 서울은 맑음...)" --from Claude

[IPC]      WM_COPYDATA → AgentBotActor.Receive<TerminalSentToBot>
[Bot]      _delegationMode == true && DONE(...) → 단축경로 (후속 #5)

[UI]       HandleAgentLoopResult → AddBotMessage → SpeakBotMessageIfEnabled
```

## 후속 수정 #10 — 위임 모드 TTS 가 발동 안 함 (IPC 가 DONE 래퍼 미리 벗김) (2026-05-12)

### Operator 보고

> 3번까지 잘됨... 4번에서 음성출력이 안됨 원인파악
> ChatBot UI 창에서는 DONE 찍히는것을 봄

후속 #9 적용 후, 위임 모드에서 사용자 발화가 Claude 에 직접 전달되는 것(1~3단계)은 확인.
Claude 가 답변 후 `bot-chat "DONE(...)" --from Claude` 콜백을 보내고 그게 ChatBot 창에 보여
지는 것까지 확인. 그런데 **4단계 음성 출력(TTS)** 이 발동 안 함.

### 진단 (운영 로그 + 코드 트레이스)

`app-log.txt` 추적 결과:
```
[02:36:20.470] [IPC] WM_COPYDATA 수신 | {"command":"bot-chat","from":"Claude","message":"DONE(...)"}
[02:36:20.471] [IPC] bot-chat from=Claude, len=102
… 그 다음 [Bot] Delegation DONE shortcut → speak inner 라인이 단 한 번도 안 보임 …
```

즉 IPC 가 도착하긴 했는데 actor 단축경로는 발동 안 함. 코드 트레이스로 원인 확인 —
`MainWindow.HandleBotChat:806-807` 에서:
```csharp
var peerName = doneFrom ?? from;
var payload  = doneMsg ?? message;   // ← DONE() 래퍼 벗긴 inner 텍스트
...
bot.Tell(new TerminalSentToBot(peerName, payload));
```

IPC 핸들러는 UI 표시 경로 (`ReceiveExternalChat`) 를 위해 DONE 래퍼를 정규식으로 미리
벗기는데, **같은 inner 텍스트가 actor 한테 forward** 됨. Actor 의 후속 #5 단축경로는
`TryExtractDoneContent` 가 `DONE(...)` prefix 를 보고 매칭하는데 이미 벗겨진 inner 만
오니까 매번 null 반환 → 단축경로 fall-through → 일반 continuation 경로로 가는데 위임
모드에서는 LLM agent loop wiring 이 없어서 (후속 #9 가 SendThroughAiToolLoopAsync 우회)
조용히 무시됨.

요약: **IPC 레이어와 actor 레이어가 DONE 래퍼에 대한 가정이 달랐음.** IPC 는 벗겨 보냈고
actor 는 래핑된 채 올 거라 기대했음.

### 수정 — 1파일 / +20 / -22 라인

`Project/ZeroCommon/Actors/AgentBotActor.cs` — `Receive<TerminalSentToBot>` 의 위임 모드 분기 재작성:

```csharp
// BEFORE: wrapper 필수 — 벗겨진 inner 텍스트는 fall-through
var doneInner = TryExtractDoneContent(msg.Text);
if (doneInner is not null) { ... fire OnResult ... return; }
// AFTER: wrapper 있으면 그걸 inner 로, 없으면 raw text 그대로
var spoken = (TryExtractDoneContent(msg.Text) ?? msg.Text ?? "").Trim();
if (spoken.Length > 0)
{
    if (IsHandshakeAck(spoken)) { 조용히 무시; return; }
    _agentLoopOnResult?.Invoke(new AgentLoopResult(true, spoken, 0, 0));
    return;
}
```

핵심 변화: **위임 모드에서 active peer 가 보낸 모든 비빈 텍스트는 TTS 로 갑니다** (handshake-ack
제외). DONE 래퍼는 있어도 되고 없어도 됨. 한 줄로 "위임 모드 = Claude 의 모든 reply 를 그대로
음성으로" 라는 operator 의 의도가 코드에 반영됨.
