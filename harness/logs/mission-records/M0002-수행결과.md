### 5. 터미널 탭 — 탭전환 버튼 제거 → DETACH/REDOCK 기능

operator가 지적한 "우측 탭 전환 버튼은 무의미"는 실제로는 AvalonDock의
LayoutDocumentPane 기본 chevron(숨겨진 탭 목록 메뉴)이다. 그 chevron 자체는
AvalonDock의 Vs2013Dark 테마 내부 리소스라 직접 제거하면 dock manager 동작에
영향을 주는 리스크가 있어, **별도 explicit DETACH/REDOCK 버튼을 CONSOLE 헤더
스트립**(`+` 버튼 좌측)에 두는 방향으로 결정.

| 버튼 | 위치 | 동작 |
|---|---|---|
| `btnDetachTab` "⤢ DETACH" | CONSOLE 스트립 우측 | 활성 탭의 `LayoutDocument.Float()` 호출 — 새 윈도우로 분리 |
| `btnRedockAll` "⤡ REDOCK" | DETACH 좌측 | 모든 분리된 탭(`IsFloating == true`)에 `Dock()` 호출 — 일괄 복귀. 분리된 탭이 없으면 disable |

핸들러는 MainWindow.xaml.cs에 `OnDetachActiveTabClick`, `OnRedockAllClick`,
`UpdateRedockButtonState` 신설. `OnLayoutRootUpdated`(layout 변경 시 호출되는
기존 콜백)에 `UpdateRedockButtonState` 호출을 추가해 드래그-드롭으로 분리/도킹된
경우에도 REDOCK 버튼 상태가 자동 동기화된다.

기존 우클릭 컨텍스트 메뉴 "Float / Dock"은 그대로 유지(이중 진입점) — 탭 단위로
선택적으로 doc하고 싶을 때 사용 가능.

### AvalonDock chevron 처리에 대해

operator의 표현 "우측에 버튼이 하나있는데 누르면 탭전환이 있음"은 AvalonDock의
LayoutDocumentPane의 chevron 메뉴 버튼을 가리킨다(많은 탭이 표시 영역을 넘으면
드롭다운으로 모두 보여주는 기능). 의미는 사실 "탭 전환"이 아니라 "탭 목록"이지만
operator의 사용 경험상 redundant하다는 평가는 동일 — 클릭 가능한 탭이 이미 있는
상황에서 별도 진입점이 가치를 주지 않는다.

이 chevron 자체를 제거하려면 AvalonDock의 Vs2013Dark 테마 ResourceDictionary
을 통해 `LayoutDocumentPaneControlStyle`을 override해야 하는데, AvalonDock 4.72의
internal API 의존성이 높아 NuGet 업그레이드 시 깨질 위험이 있다. **기능 동등성**을
최우선으로 두고 — operator가 원하는 핵심 기능은 "버튼 하나로 탭을 새창으로
분리하고 다시 복원"이므로 — chevron은 그대로 두고 CONSOLE 스트립에 explicit한
DETACH/REDOCK 버튼을 추가하는 방식으로 충족했다.

만약 향후 chevron이 시각적으로 거슬린다는 추가 피드백이 오면 별도 미션(M0003 등)으로
AvalonDock 테마 override 작업을 검토하면 된다.

### REDOCK이 "현재 탭만"이 아닌 "전부"인 이유

1차 설계는 "현재 활성(focused) 탭이 floating이면 그걸 dock"이었지만 — 메인
윈도우의 활성 탭은 항상 docked이고, floating 탭이 활성이려면 floating window가
focused여야 한다. 메인 윈도우의 CONSOLE 스트립 버튼을 클릭하는 그 순간 메인
윈도우가 focus를 가져가서 floating window는 비활성이 된다. 그래서 "현재 탭"
기준이면 REDOCK 버튼은 사실상 한 번도 동작 가능 상태가 안 된다.

대안:
- **(A)** 모든 floating 탭을 한 번에 dock — 단순. 사용자가 분리한 탭이 1개일 때
  자연스럽고, 여러 개라면 의도적으로 한꺼번에 정리하는 액션이 됨.
- (B) 드롭다운으로 floating 탭 목록 보여주고 선택 — 추가 UI 필요, 1탭 케이스에선
  과잉.

(A)를 선택했다. floating 탭이 0개면 버튼은 자동 disable이라 잘못 누를 일도 없다.

### 사후 수정 — REDOCK이 동작하지 않는 회귀

operator의 GUI 검증에서 DETACH는 정상이나 **REDOCK 클릭 시 floating 창이 메인으로
복귀하지 않는** 회귀가 보고됐다. 원인 분석:

1. 1차 구현은 `LayoutContent.Dock()` API에 의존했다.
2. 그러나 AgentZero Lite의 dock manager는 `OnLayoutRootUpdated` 핸들러에서
   "터미널 doc pane이 비어 prune되면 새 인스턴스로 재생성"하는 동작을 한다
   (line 2331 부근). float이 일어나는 순간 원래의 `terminalDocPane`은 자식이
   비어 AvalonDock이 자동 prune → handler가 새 LayoutDocumentPane을 만들어
   삽입한다.
3. 결과: floating 상태의 LayoutDocument는 `PreviousContainer`로 **이미 분리된
   원본 pane(orphan)** 을 가리키고 있어, `Dock()`이 호출돼도 invariant 검사에서
   silent no-op로 떨어진다(예외 throw도 없음).

수정 방향: 코드베이스에 이미 검증된 패턴 사용 — group 전환 로직(line ~1645)에서
쓰는 `parent.RemoveChild(doc)` + `target.Children.Add(doc)` 직접 이동. 이 방식은
`PreviousContainer` 상태와 무관하게 강제 재배치하므로 prune-replace 사이클의
영향을 받지 않는다.

`OnRedockAllClick` 본체를 다음과 같이 교체:

```csharp
var floating = _consoleTabs
    .Where(t => t.Document is { IsFloating: true })
    .Select(t => t.Document!)
    .ToArray();
if (floating.Length == 0) return;

var target = GetActiveDocumentPane();    // 기존 헬퍼 — orphan-safe
foreach (var doc in floating)
{
    if (doc.Parent is AvalonDock.Layout.ILayoutContainer parent)
        parent.RemoveChild(doc);
    target.Children.Add(doc);
}
last.IsActive = true; last.IsSelected = true;   // 마지막 복귀 탭을 활성화
```

빌드 재확인: 0 errors. 이번 수정의 lesson — 이 코드베이스에서 AvalonDock의
`LayoutContent.Float()/Dock()` API는 단방향(Float만) 신뢰 가능하고, redock은
명시적인 children 조작으로 구현해야 한다. 향후 `OnDocTabDock`(우클릭 컨텍스트
메뉴 버전)도 동일 패턴으로 정리할 가치가 있으나, operator가 회귀를 보고하지
않은 한 변경하지 않음(리스크/리워드 균형).

### 사후 수정 2 — REDOCK 위치를 floating 윈도우 내부로 이동 (UX 정합성)

operator의 추가 피드백:

> "복원 기능은 새창의 윈도우창에 있는게 좋을것같음 ... 기존기능 이곳으로 돌아와
> 느낌( 새창복원 기능은 여기제거)... 이 느낌의 UX를 이해하고 적용할것"

분석:

메인 윈도우의 CONSOLE 스트립에 REDOCK 버튼을 둔 1차 설계는 멘탈 모델과 어긋났다.
사용자가 "돌아간다"는 감각을 느끼는 시점은 **이미 floating 윈도우에서 작업 중일
때**인데, 그 순간 메인 윈도우는 백그라운드에 있고 REDOCK 버튼은 보이지 않는다.
alt-tab → 메인 활성화 → 버튼 찾기 → 클릭 — 인지 비용이 4단계.

올바른 위치는 floating 윈도우 자체. operator는 또한 AvalonDock가 floating 윈도우
탭 우측에 두는 chevron(다른 탭 목록 드롭다운)을 "포커싱 방해"로 지목하고 그
자리에 REDOCK이 들어갔으면 좋겠다고 했다.

조치:

1. **메인 윈도우 REDOCK 버튼 제거** — `MainWindow.xaml`의 `btnRedockAll` 마크업
   삭제. `OnRedockAllClick` / `UpdateRedockButtonState` 핸들러도 제거. CONSOLE
   스트립엔 DETACH 버튼만 남는다 (DETACH는 메인에 있는 게 맞다 — "지금 보고 있는
   탭을 떼낸다").

2. **각 탭별 REDOCK 스트립 — TerminalHost row 0**

   `ConsoleTabInfo`에 `RedockStrip: Border?` 필드 추가. 탭 생성 시점
   (`CreateConsoleTab` ~line 1751)에 `termHost`(Grid)에 RowDefinitions 추가:
   - Row 0: `Auto` — REDOCK 스트립 (`Visibility=Collapsed`이 기본, 0px 차지 안 함)
   - Row 1: `*` — 터미널 + WedgeBanner (`Grid.Row=1` 명시)

   REDOCK 스트립은 DockPanel 안에 라벨("↩  Floating window — click to dock back
   to main") + 버튼("↩ DOCK BACK") 두 요소. 다크 시안 톤(`#00FFF0` foreground,
   `#0A1A24` background, `#006B80` 보더). 클릭 시 `RedockOneTab(tab)` 호출.

   이 스트립은 `LayoutDocument.Content` (== termHost)에 박혀 있어 doc이 floating
   윈도우로 옮겨질 때 함께 이동한다 — floating 윈도우에서 자연스럽게 보인다.

3. **Visibility 동기화** — `OnLayoutRootUpdated` (AvalonDock layout 변경 시
   호출되는 기존 콜백)의 끝에 `RefreshAllRedockStripVisibility()` 호출 추가.
   모든 그룹의 모든 탭을 순회하며 `tab.Document.IsFloating` 기준으로
   `tab.RedockStrip.Visibility`를 토글한다. 따라서:
   - DETACH 클릭 → Float 발생 → OnLayoutRootUpdated 발화 → 스트립 Visible
   - REDOCK 클릭 / 우클릭 Dock / 드래그 도킹 → OnLayoutRootUpdated 발화 → 스트립
     Collapsed

4. **Redock 로직** — `RedockOneTab(ConsoleTabInfo tab)` 신규. 단일 탭만
   대상으로 하므로 사후수정 1의 일괄 처리보다 더 정합적이다(여러 floating
   윈도우에 흩어진 탭 중 사용자가 "이 창만" 복귀시킬 수 있음). 동일한
   RemoveChild + Children.Add 패턴.

### 의도적으로 안 한 것 — chevron 제거

AvalonDock floating 윈도우의 탭 strip 우측에 그려지는 chevron(드롭다운
컨텍스트 메뉴)은 Vs2013Dark 테마의 internal 리소스라 override가 NuGet 4.72
의존성에 잠긴 surgery가 된다(향후 NuGet 업그레이드 시 깨짐). 새 REDOCK 스트립이
탭 컨텐츠 영역 최상단에 강조 색으로 배치되어 발견성은 chevron보다 훨씬 높으므로
chevron 자체는 무해한 cosmetic 잔재로 남긴다. operator가 명시적 제거를
요청하면 별도 미션(M0003 후보)으로 처리.

### 사후 수정 3 — redock 후 빈 floating 윈도우 잔존 fix

operator의 추가 피드백:

> "잘돌아감 하지만 사소한 버그 fix 돌아간뒤.. 새창윈도우가 남아있음..이것도
> 제거해야 클린할듯함.. 터미널창 이동은 잘됨 새창모드에서 터미널을 가지는
> 부모 창이남은 이슈"

증상: REDOCK 클릭 → 탭은 메인으로 잘 이동 → 그러나 비어버린 floating 윈도우
shell은 화면에 그대로 남는다. AvalonDock가 마지막 LayoutDocument이 floating
윈도우에서 제거돼도 호스트 Window를 자동 close하지 않는 동작 때문.

조치: `CloseEmptyFloatingWindows()` 헬퍼 신설.

```csharp
private void CloseEmptyFloatingWindows()
{
    var fws = dockManager.FloatingWindows?.ToArray();
    if (fws is null) return;
    foreach (var fw in fws)
    {
        bool empty = fw.Model is not AvalonDock.Layout.ILayoutElement modelEl
                  || !ContainsLayoutDocument(modelEl);
        if (!empty) continue;
        fw.Close();
    }
}

private static bool ContainsLayoutDocument(AvalonDock.Layout.ILayoutElement el)
{
    if (el is AvalonDock.Layout.LayoutDocument) return true;
    if (el is AvalonDock.Layout.ILayoutContainer c)
        foreach (var child in c.Children)
            if (ContainsLayoutDocument(child)) return true;
    return false;
}
```

`Descendents()` 확장이 AvalonDock 4.72의 모든 Layout 타입에 노출돼 있지 않아
재귀 walker(`ContainsLayoutDocument`)는 코드베이스의 기존 패턴(`FindDocumentPane`,
line ~1826)을 그대로 따랐다 — `ILayoutContainer.Children` 재귀.

호출 지점 두 곳:

1. `RedockOneTab` 끝 — REDOCK 클릭 직후 즉시 sweep. 사용자 체감 응답 속도 ↑.
2. `OnLayoutRootUpdated` 끝 — safety net. floating 윈도우 안에서 X로 마지막
   탭을 닫거나, drag-drop 도중 문제로 빈 shell이 남는 경로도 함께 커버.

빌드 재확인: 0 errors. 이제 REDOCK 후 floating 윈도우가 깔끔하게 정리된다.
