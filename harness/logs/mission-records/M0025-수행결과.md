# M0025 수행결과 — Agent Band 플러그인

## 실행 요약

미션 brief = **방금 도입한 음악(AST AudioSet) 모델을 WebDev 플러그인으로 끌어내,
재생 중인 사운드의 가수/악기를 인식해 매칭되는 연주자 스프라이트를 무대 위에
동적으로 소환·애니메이트하기**.

설계 결정의 핵심 갈래는 두 가지였다.

1. **JS 라이브러리 채택** — brief는 "주로 게임쪽 2D canvas 권장"을 명시했고
   PixiJS가 자연스러운 후보였지만, `Wasm/` 샌드박스는 **오프라인 무네트워크
   원칙** (M0005 에서 정의) 을 지킨다. PixiJS를 vendor 하면 ~500KB 추가 +
   plugin 입장에서 외부 의존성이 발생한다. 단순 sprite blit + 4-frame 애니
   + 64-bin spectrum 만 필요하기 때문에 **vanilla Canvas 2D** 로 직접 구현.
   결과적으로 외부 라이브러리 0개, 라이브 60 Hz 렌더 + 1.5 s AST tick 합성.

2.5 **배포 방식** — 첫 라운드는 `Wasm/agent-band/` 내장 플러그인으로 두었으나,
   operator 가 "WebDev 의 Git URL 기반 플러그인 설치 흐름을 활용하라" 요구.
   `WebDevPluginInstaller.InstallFromGitUrlAsync` 가 이미 GitHub `tree` URL
   에서 manifest.json 검증 → Trees API 폴더 enumerate → raw URL 일괄 다운로드
   → 무결성 체크 → `%LOCALAPPDATA%/.../Wasm/plugins/{id}/` 로 이동 까지 다
   해줌. 따라서 두 가지만 추가하면 됨:
   - 폴더 위치를 `Project/Plugins/agent-band/` 로 이동 (내장이 아닌 순수 사용자 플러그인
     으로 정체성 통일 — 미래의 AZ 본 릴리스에서도 중복 노출 없음)
   - `manifest.json` 추가 (id/name/entry/version/description)
   - `<script src="../common/zero-bridge.js">` 가 user-plugin 경로
     (`plugin.local/agent-band/`) 기준으로는 빈 폴더를 가리키므로,
     **`https://zero.local/common/zero-bridge.js`** 절대 URL 로 변경.
     WebView2 가 두 virtual host (`zero.local` 내장 / `plugin.local` 사용자)
     를 같은 인스턴스에 매핑하므로 cross-origin `<script>` 로딩이 동작.
     플러그인은 항상 호스트의 최신 브리지 표면(`music.*`, `note.*`, …) 을
     자동 픽업, 스냅샷이 stale 될 일 없음.

3. **AST 추론 루프 위치** — Settings/Music 탭이 이미 동일한 loopback +
   sliding window + AST inference 루프를 갖고 있다 (M0023 도입). 이 코드를
   WebView2 브리지가 호출할 수 있는 형태로 재포장해야 하는데, 두 옵션이 있었다:
   - (a) `SettingsPanel.Music.cs` 의 핵심 로직을 ZeroCommon 으로 추출 후 양쪽이 공유
   - (b) `WebDevHost` 에 `note.*` 와 같은 패턴으로 새 `music.*` 라이브 루프 추가

   브리지 패턴(b)을 채택. 이유: Settings UI 의 직접 paint 경로 (Dispatcher,
   WPF Rectangle 등) 와 분리되어 있어 추출 비용이 크고, voice-note (M0007)
   가 이미 같은 WebDevHost 안에서 동일한 패턴으로 자기 라이프사이클을
   소유한다. 새 `WebDevHost.Music.cs` partial 이 그 패턴을 그대로 따른다.

- "상단은 잘려도 되나 연주자가 잘 배치" → `drawStageBg()` 가 cover-fit 하면서

  정렬, 스프라이트 종횡비 보존하면서 슬롯 폭의 100% 또는 슬롯 높이 62%

  순. 비어 있는 위치는 압축되지 않고 사라지므로 항상 무대가 균형 잡힘.

## 평가 (3축)

### 1. 코드 안전성 (Code Safety)

**등급: A**

- 호스트 측 — 기존 `note.*` 라이프사이클 패턴 그대로 답습. `_musicLock`
  세마포어로 Start/Stop 직렬화, 이벤트 핸들러 -= 깨끗하게 처리, classifier
  세션 재사용으로 ONNX 더블 로드 방지. 단일 인스턴스 호스트라 race
  표면 없음.
- TearDownMusicLocked 가 CancellationToken cancel → Task.Wait(500) → handlers
  detach → Stop/Dispose 순서로 진행. 기존 voice-note 패턴과 동일.
- 플러그인 측 — `beforeunload` 에서 stop 호출. `window.zero?.music?.stop()`
  optional chaining 으로 호스트 disconnect 시 silent fail.
- 빌드 경고 신규 0개. 기존 7개 경고(other) 그대로.
- 테스트: 헤드리스 xUnit **294 pass / 24 skip / 0 fail**.

리스크: classifier 인스턴스를 stop 사이 유지하지만 `MusicSettings` 가
바뀌면 stale — `MusicSettingsStore.Save()` 후 invalidation 훅이 없어
다음 Start 가 옛 모델 경로로 동작할 수 있음. Settings 탭은 이미 같은
패턴 (Save 시 classifier null-out) 을 갖고 있으나 WebDevHost 는 Settings
저장 이벤트를 구독하지 않음. 후속에서 보완하거나 plugin Settings 화면을
열기 직전 host classifier 무효화 훅 추가가 안전.

### 2. 아키텍처 정합성 (Architecture Fit)

**등급: A**

- **3계층 분리 유지**: 액터 0개 변경. 음악 추론은 voice-note 와 동급의
  WebDevHost 인스턴스 상태로 표현 — 액터 토폴로지는 그대로.
- **ZeroCommon vs WPF**: 호스트 측 코드 전부 `AgentZeroWpf` 안에 머묾.
  NAudio WASAPI loopback + ONNX 추론은 WPF 가 아닌 .NET 서비스라 ZeroCommon
  으로 이동시킬 수도 있으나, `_uiDispatcher` 마샬링 컨텍스트는 WebDev
  panel 이 소유 — 현 위치가 자연스러움.
- **IZeroBrowser**: 의도적으로 확장하지 않음. music.* 는 voice-note 와
  마찬가지로 `WebDevHost` 의 추가 메소드 + Bridge dispatch 케이스로만
  구현 — IZeroBrowser 는 chat/voice 의 "공식" 표면만 유지. note.* 와 정확히
  같은 escape hatch 패턴.
- **brief 의 "주로 게임쪽 2D canvas 라이브러리 권장"** 을 vanilla 로 가는
  결정 — `Wasm/` 가 오프라인 무네트워크 원칙(M0005) 이라 PixiJS 를
  vendor 하면 ~500KB 가 모든 새 플러그인이 짊어져야 할 비용이 됨. 이번 build
  에는 외부 의존성 0, 다음 플러그인이 더 무거운 게임 라이브러리를 정말
  필요로 하면 `common/` 에 PixiJS 1회 vendor 후 모든 플러그인이 공유하는
  형태로 확장 가능.

### 3. 테스트 가능성 (Testability)

**등급: B+**

- 호스트 측: AST 모델 로딩 + ONNX 세션은 실 환경에서만 의미 있는
  통합 테스트. 단위 테스트 적합 영역은 두 곳:
  - `WebDevHost.Music.MusicTickInfo` 직렬화 (post-event JSON shape)
  - rolling buffer trim 로직 (현재 inline)

  현재 ZeroCommon.Tests 가 헤드리스이므로 WebDevHost 단위 테스트는 새
  AgentTest 표적이 적절. 이번 라운드는 추가하지 않음.

- 플러그인 측: `labelToPerformer()` 의 regex 매핑은 pure function 으로
  순수 단위 테스트 대상이지만, WebView2 sandbox 안의 JS 라 unit-test 인프라
  없음. 후속에서 `Wasm/common/` 에 미니 test runner 가 들어오면 회귀
  테스트 hook 가능.

- **설치 + 수동 smoke path** (사용자 측):
  1. WebDev 메뉴 → 플러그인 설치 UI → Git URL 입력:

  2. 인스톨 완료 후 좌측 2뎁스에 `Agent Band` 항목 등장
  3. 클릭 → Start → model-missing 시 status 가 빨갛게 안내, 그 외에는 LIVE
  4. Spotify/YouTube 등 음악 재생
  5. ~3 s 후 label strip 에 AudioSet 분류 표시, 매칭 sprite 가 stage 에 fade-in
  6. 활성 sprite 는 4-frame 'play' 애니로 사이클, 미활성은 4-frame 'idle'
  7. 소리 끊기면 ~6 s 후 sprite fade-out
  8. Spectrum bar 가 카타르 그라데이션으로 항상 흐름
