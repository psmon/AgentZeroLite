# 수행 요약

CMD / PowerShell 5 / PowerShell 7 CLI 정의에 **Remote (SSH) 모드** 를 도입했습니다.
사용자가 CLI Definition 다이얼로그에서 `Remote` 체크박스를 켜고 Host/User/인증 방식
(Public Key `.pem` 또는 Password) 을 입력하면, 해당 정의로 새 터미널 탭을 열
때마다 셸 startup 명령이 자동으로 `ssh user@host` 로 치환됩니다. 비밀번호는
**Windows DPAPI (CurrentUser scope)** 으로 암호화되어 SQLite 에 저장되며,
평문은 디스크 어디에도 남지 않습니다. 터미널이 실제로 spawn 되는 순간만 한 번
복호화되어 클립보드로 흘러나가 — OpenSSH 가 argv 로 비밀번호를 받지 않는다는
제약을 가장 가벼운 방식으로 우회합니다.

## 비고

### 의도적으로 보류한 항목 (후속 미션 후보)

1. **Password 모드 자동 입력 (ssh-askpass 또는 plink)**.
   현재는 클립보드 + 사용자 paste 로 처리합니다. 자동화하려면:
   - `SSH_ASKPASS` 환경변수 + 헬퍼 exe 패턴: Windows OpenSSH 에서 ASKPASS 가
     까다롭고, 헬퍼가 plaintext 를 stdout 으로 흘리는 순간 process tree 가 보임.
   - PuTTY `plink -pw` 도입: 외부 의존, 라이선스 + ConPTY 호환성 추가 검토 필요.
   둘 다 1 차 범위 밖. 클립보드 + 토스트는 가장 검증된 패턴이라 v1 로 충분.

2. **SSH 호스트 키 fingerprint 사전 등록 / 검증**.
   현재는 OpenSSH 가 첫 연결에서 `known_hosts` 에 추가할지 사용자에게 묻습니다.
   엔터프라이즈 환경에서는 fingerprint 를 CliDefinition 에 같이 저장해 mismatch
   시 경고하는 패턴이 있을 수 있음 — 보안 강화 후속.

3. **Connection profile 다중화**.
   현재는 CliDefinition 1 개당 SSH 설정 1 개. 동일 셸로 여러 호스트를 빠르게
   전환하려면 SshProfile 별도 테이블 + 한 정의에 다중 프로파일 매핑이 필요.
   사용자가 그런 패턴이 필요하다고 신호하면 후속에서.

4. **Settings UI 에서 비밀번호 확인 표시**.
   현재 편집 시 PasswordBox 는 비워두면 기존 ciphertext 유지. "Password
   stored ✓" 같은 정적 indicator 라벨이 있으면 더 친절. 단순 UX 라운드.

5. **PowerShell `-Command` 인자 escape 강화**.
   현재 host/user 에 공백이나 특수문자가 들어가면 PowerShell 의 -Command 가 깨질
   수 있음. 일반적인 SSH 호스트명 / 사용자명은 ASCII alnum + `-` / `.` / `@` 라
   1 차에선 단순 concatenation. 후속에서 quoting 정책 추가.

### Windows DPAPI 를 고른 이유

- **OS 가 키를 관리** — AgentZero 가 마스터 키를 들고 있지 않으므로 앱 바이너리
  유출만으로 ciphertext 가 깨지지 않음.
- **CurrentUser scope** — SQLite 파일을 다른 사용자 / 머신으로 복사해도 복호화
  실패. 가장 일반적인 "DB 백업 분실 → 평문 노출" 시나리오를 차단.
- **추가 NuGet 패키지 없음** — `System.Security.Cryptography.ProtectedData` 는
  `Microsoft.WindowsDesktop.App.Ref` (WPF SDK) 에 이미 포함. CLAUDE.md 의 "
  System.Speech 10.0.0-preview" 와 같은 결.
- **AES + key derivation 자체 구현 회피** — 보안 코드를 직접 쓰지 않는 게 최선.

대안으로 검토했던 `DataProtectionProvider` (`Microsoft.AspNetCore.DataProtection`)
은 ASP.NET Core 가 호스팅하는 키링 구조라 데스크탑 앱엔 과한 인프라.

### 사용 흐름 (사용자 관점)

1. Settings → CLI Definitions → Add (또는 기존 정의 Edit)
2. Name / ExePath 입력 (cmd.exe / powershell.exe / pwsh.exe 중 하나)
3. **Remote (SSH)** 체크박스 켜기 → Host / User / Auth 패널이 펼쳐짐
4. **Public key (.pem)** 선택 시 → Browse… 로 `.pem` 선택
5. **Password** 선택 시 → PasswordBox 에 입력. 저장 시점에 DPAPI 로 암호화되며
   PasswordBox 는 즉시 Clear.
6. OK → SQLite 에 6 개 필드 함께 저장.
7. (Optional) 앱 재시작 후에도 DB 에 저장된 정의가 그대로 복원됨.
8. CLI 탭 + 메뉴에서 해당 정의 선택 → 새 탭이 열리며 셸 startup 이
   `ssh user@host` (또는 `ssh -i "<pem>" user@host`) 으로 자동 치환됨.
9. Password 모드면 클립보드에 평문이 1 회 복사됨 + 로그에 `[Ssh] Password
   copied to clipboard` 기록. 사용자가 ssh 프롬프트에서 paste.

전제: OpenSSH 클라이언트가 PATH 에 있어야 합니다 (Windows 10 1809+ 기본 포함).
없으면 ssh: command not found 에러가 PTY 에 그대로 노출됩니다.

`Project/AgentZeroWpf/bin/Debug/net10.0-windows/logs/app-log.txt` 의 752 / 812 /
881 라인에 매번 다음이 정상 기록됨:

```
[21:13:13.822] [Ssh] Password copied to clipboard for tab "Hermes" — paste at the ssh prompt.
[21:13:13.823] [CLI] ConPTY 터미널 생성 (lazy): cmd /c "...cmd.exe /K ssh psmon@192.168.0.64"
```

즉 **저장은 정상**:

1. DPAPI 암호화 라운드트립 ✓ — `Unprotect` 이 null/throw 가 아니라 정상 plaintext 반환
2. SQLite 영속 ✓ — 매 탭 활성화마다 ciphertext 가 그대로 디스크에서 살아 돌아옴
3. 클립보드 set ✓ — `Clipboard.SetText` 가 던지지 않고 통과

진짜 문제는 **사용자 멘탈 모델 불일치**:

| 사용자 기대 | 실제 v1 동작 |
|---|---|
| "Save 했으니 ssh 가 알아서 로그인" | ssh 가 여전히 `Password:` 프롬프트 출력 |
| "비번이 자동 적용됨" | 사용자가 직접 클립보드를 paste 해야 함 |
| "비번 저장 = 자동 입력" | 비번 저장 ≠ 자동 입력 (수동 paste 필요) |

OpenSSH 는 디자인상 argv (`ssh -pw …` 같은 플래그) 로 비밀번호를 받지 않고
오직 `getpass()` 의 raw-mode tty 입력만 받는다. v1 의 "clipboard 안내" 가 가장
손쉬운 회피로 보였지만, "비번 저장이 안되" 라는 사용자 체감은 정확. 자동
주입이 안 들어가면 "저장의 가치" 가 사라진다.

또한 ConPTY 의 paste shortcut 은 ssh 가 hidden-echo 모드에서 입력 받는 동안
키 시퀀스 해석이 미묘해서 일반 paste 가 깔끔하게 안 들어가는 케이스가 있음.
설사 들어가더라도 사용자가 "Password:" 출력을 인지하고 paste 키를 누르는 동선
자체가 자동화 약속과 배치된다.

### 수정 (자동 입력으로 전환)

**Auto-fill watcher 패턴** — `Project/AgentZeroWpf/Services/SshPasswordAutofill.cs` 신규:

- 생성 시 `(ConsoleTabInfo tab, string plaintext)` 받음
- `DispatcherTimer 150ms tick` 으로 `tab.Session` 이 lazy 생성되기를 폴링
- Session 부착되면 즉시:
  - `OutputReceived` 이벤트 구독
  - 부착 직후 `GetConsoleText()` 한 번 스캔 (subscription 전에 이미
    프롬프트가 나왔을 수 있는 race 대응)
- `"assword:"` 부분 문자열을 케이스-인센서티브로 매칭 (`"Password:"`,
  `"password:"`, `"[user@host]'s password:"` 모두 잡힘 — VT escape 가
  중간에 끼어도 `assword:` 자체는 깨지지 않음)
- 매칭 시 `session.WriteAndSubmit(plaintext)` — 50ms 간격 두고 `\r` 까지
  보내는 패턴 (다른 TUI 자동입력과 동일)
- 1 회 배달 후 즉시 `Dispose`: 타이머 정지 + 이벤트 unsubscribe + plaintext
  필드 `= null` (best-effort zero-out)
- 2 분 expire — 사용자가 ssh 안 한다거나, 프롬프트 형식이 매칭 안 되어도
  plaintext 가 무한히 메모리에 남지 않도록 보장. 만료 시 `[Ssh] Auto-fill
  expired before "assword:" prompt arrived` 라고 로그.

**InitializeTerminal 갱신**:

```csharp
if (!string.IsNullOrEmpty(tab.EncryptedPasswordForLaunch))
{
    var plaintext = DpapiSecretProtector.Unprotect(tab.EncryptedPasswordForLaunch);
    if (!string.IsNullOrEmpty(plaintext))
    {
        _ = new SshPasswordAutofill(tab, plaintext);             // 주 경로
        try { System.Windows.Clipboard.SetText(plaintext); } catch { } // 폴백
        AppLogger.Log($"[Ssh] Auto-fill armed for tab \"{tab.Title}\" …");
    }
}
```

클립보드 폴백을 유지하는 이유: (a) 매칭 needle (`assword:`) 와 다른 비표준
프롬프트 (랩퍼 스크립트 등) 에서 autofill 이 놓치는 경우 사용자가 paste 로
복구 가능, (b) ssh 첫 connection 시 `Are you sure you want to continue
connecting (yes/no/[fingerprint])?` 같은 호스트 키 확인이 먼저 뜨면 autofill
은 그 때 `assword:` 매칭이 안되므로 그냥 대기 → 사용자가 yes 입력 후 실제
프롬프트가 오면 그때 autofill 발동. 사이드 fallback 으로 클립보드가 있으면
어쨌든 한 번은 paste 가능.

### 보안 trade-off

| 비교 | v1 (clipboard only) | v2 (auto-fill primary + clipboard fallback) |
|---|---|---|
| Plaintext 디스크 잔존 | 없음 | 없음 |
| Plaintext 메모리 잔존 시간 | 클립보드 수명 (사용자 다른 복사 전까지 ~수분) | autofill object 수명 (최대 2 분, 평균 1 초) |
| Plaintext zero-out | OS 클립보드 정책에 의존 | autofill 의 finally 에서 `_plaintext = null` (GC 가 처리) |
| 다른 프로세스가 훔칠 surface | 클립보드를 보는 모든 앱 (Win32 클립보드 API) | autofill 인스턴스 + 클립보드 둘 다 |
| 자동화 충실도 | 사용자 paste 동작 필요 | 자동 |

clipboard 폴백을 제거하면 더 안전 (autofill 만 남으면 메모리 노출이 더 짧음).
하지만 한 라운드만에 폴백 제거는 회귀 위험이 있어 우선 v2 = 자동입력 + 클립보드
폴백, 사용자 검증 후 폴백 제거를 후속 미션 (M0022+) 으로 둠.

- **PromptNeedle 미스매치**: 랩퍼 스크립트가 `Enter password: ` 같은 다른 형식
  을 쓰는 경우. 현재 `assword:` 매칭이라 대부분 잡히지만 만약 안 잡히면
  로그에 `Auto-fill expired` 가 나오고 클립보드 폴백 가능.
- **이중 ssh / sudo**: 원격 ssh 안에서 다시 sudo 같은 게 `Password:` 를 출력하면

  재무장 — 의도된 동작이지만 UX 가 아쉬우면 후속에서 ssh 명령 재실행 감지
  로직 추가 가능.

## 후속 수정 #2 — autofill 진단 + delete 오류 처리

### Operator 보고 (verbatim)

> 값이있나 확인? Hermes 에저장했는데 여전히 비번 요구함 체크해
> ... (이후) 로그확인해죠 안됨 그리고 delete 도 오류남 hermes 설정으로 테스트

### 진단 (DB + log 동시 확인)

**A) Hermes 값 — DB 확인 결과 100% 정상.** Microsoft.Data.Sqlite 로 SQLite 직접
인스펙트 결과:

```
ID=12 Name=Hermes Exe=cmd.exe Remote=1 Host=192.168.0.64 User=psmon
Auth=Password PwLen=308 PwHead=AQAAANCMnd8BFdERjHoAwE/Cl+sBAAAA
```

- 308 자 base64 ciphertext
- `AQAAANCMnd8BFdER` — **DPAPI CurrentUser blob 의 표준 매직 헤더**
- Migration `20260518100000_AddCliDefinitionRemoteShell` 적용됨
- 즉 저장은 완벽, "비번 저장이 안되" 라는 체감은 다른 원인

**B) autofill 이 부착됐는데 needle 매칭 0회** — `[Ssh-AF]` 진단 로그를 박은
빌드에서 다음이 드러남 (라인 451-471):

```
21:29:24.052  [Ssh-AF] ctor tab="Hermes" pwLen=7 needle="assword:" expireAfter=120s
21:29:24.382  [Ssh-AF] attached after 1 ticks (150ms)
21:29:24.386  [Ssh-AF] initial-scan consoleLen=80 match=False
              tail=<<<C:\Windows\SYSTEM32\cmd.exeC:\Windows\SYSTEM32\cmd.exe - ssh  psmon@192.168.0.64>>>
21:29:26.439  [Akka] Terminal destroyed (dock close)
```

두 가지 결정적 사실:

1. **`GetConsoleText(true)` 가 PTY stdout 이 아니라 cmd.exe 윈도우 타이틀바
   텍스트만 반환**. `cmd.exeC:\…cmd.exe - ssh psmon@…` 가 그 증거 (Windows 가
   자식 프로세스 이름으로 타이틀 두 번 업데이트한 흔적 concat). 진짜 ssh 출력
   은 `ConsoleOutputLog` 에 있어야 하는데 GetConsoleText 가 다른 view 를 본
   거였음.
2. **부착 후 2초만에 탭 destroy** — `OutputReceived` 가 한 번도 안 옴
   (`frame#N` 로그 0 개). 그 사이 ssh handshake 가 password 프롬프트까지
   못 갔거나, operator 가 답답해서 탭을 닫음.

**C) Delete 오류** — `CliTab → CliDefinition` FK 가 `OnDelete: Restrict` 라,
Hermes 정의를 참조하는 CliTab 이 살아있는 동안 `db.CliDefinitions.Remove +
SaveChanges` 가 `DbUpdateException` 으로 터짐. `OnDeleteCliDef` 에 try/catch
없어서 unhandled 예외로 사용자한테 노출됐던 게 "delete 오류" 의 정체.

### 수정

**1. `SshPasswordAutofill` v3 — 실제 PTY stream 읽기**

- `GetConsoleText` 호출 제거 → `ReadOutput(0, OutputLength)` 로 교체.
  ReadOutput 은 `ConsoleOutputLog` (StringBuilder 의 실제 stdout) 를 직접 읽음.
- `ScanCurrentOutput(source)` 헬퍼 분리 — 부착 시점에 한 번 + 매 4 frame 마다
  전체 버퍼 재스캔. needle 이 두 frame 경계에 걸려도 잡힘.
- 모든 frame 로깅 (이전엔 첫 5 + 매칭만) — 다음 시도에서 ssh 가 실제로 어떤
  텍스트를 흘리는지 한 줄씩 노출.
- `attached` 로그에 `outputLen={N}` 추가 — 부착 시점의 누적 출력량 가시화.

**2. `OnDeleteCliDef` — usage check + try/catch**

- 삭제 전 `db.CliTabs.Where(t => t.CliDefinitionId == selected.Id).ToList()`
  로 참조 중인 탭 enumerate.
- 0 개 → 기존 동작 (간단한 confirm).
- 1 개 이상 → `"Delete CLI definition 'X'? N tab(s) currently use this
  definition and will be removed as well."` 다이얼로그.
- 승인 시 `db.CliTabs.RemoveRange(referencingTabs)` 먼저, 이어서
  `db.CliDefinitions.Remove(entity)`, 그리고 `db.SaveChanges()` — 전체를
  try/catch 로 감싸 잔여 오류는 `MessageBox` 로 표시 + AppLogger 기록.
- 이전엔 unhandled 예외로 터졌던 게 이제 명확한 UI 메시지로 전달됨.

## 후속 수정 #3 — Password 모드를 plink (PuTTY) 로 전환, autofill 패턴 폐기

### Operator 보고 (verbatim)

> ssh arg 패스워드 전달하면 되는거 아님? 혹시 수동입력으로 진행함?
> ... (선택지 제시 후) 패스워드만 plink 적용

### 진단 / 설계 결정

OpenSSH (Windows 기본 `ssh.exe`) 가 `--password` / `-pw` 같은 플래그를 제공
**하지 않음** — process listing (Task Manager Command Line 열, `ps` 출력) 으로
비밀번호가 유출되는 걸 막으려는 의도된 보안 디자인. v1 (clipboard) / v2
(PTY 출력 감지 후 자동 typing) 는 그 제약을 회피하려는 우회로였지만 둘 다
race-prone (timing / needle 매칭 / PTY 출력 view 불일치).

**PuTTY 의 `plink.exe`** 는 정반대 철학으로 `-pw "..."` 를 받음 — 자동화 친화적.
trade-off: 연결 중 짧은 시간 동안 비밀번호가 process listing 에 노출됨. 단,

- 디스크 저장은 여전히 DPAPI ciphertext (디스크 유출 0)
- plaintext 는 `RebuildCliContextMenu` 의 click 클로저 → `ComposeArguments`
  결과 문자열 → ProcessStartInfo 까지만 살고 그 이후엔 cmd.exe 의 argv 라
  Windows 가 관리
- single-user desktop 환경에선 충분히 합리적

→ Password 모드만 plink 로 전환. Public key 모드는 openssh `ssh -i` 그대로.

### 수정

**1. `SshCommandBuilder.BuildSshCommand(ssh, plaintextPassword)` 시그니처 확장**

- PublicKey: `ssh -i "<key>" user@host` (변경 없음)
- Password + plaintext 있음: **`plink.exe -ssh -pw "<escaped>" user@host`**
- Password + plaintext 없음 (DPAPI 실패 등 폴백): `ssh user@host`
- `EscapePlinkPassword` 헬퍼 — 비밀번호 내 `"` 를 `\"` 로 escape (cmd.exe 의
  outer /c 경계 보호). 단, 비밀번호에 `& | < > ^ %` 같은 cmd metachar 가
  있으면 cmd 가 plink 도착 전에 해석하므로 비권장 — 다이얼로그 tip 에 명시.

**2. `ComposeArguments` 가 `plaintextPassword` 를 forward**

- 호출부가 미리 DPAPI Unprotect 해서 plaintext 를 넘기면 BuildSshCommand 가
  plink 경로 선택. PublicKey 모드는 이 인자 무시.

**3. 호출부 (Launcher) 갱신**

- `MainWindow.RebuildCliContextMenu` 의 click 핸들러:
  - menu build 시점엔 ciphertext 만 캡처 (plaintext 노출 X)
  - click 시점에 `DpapiSecretProtector.Unprotect` → `ComposeArguments(plaintextPw)`
  - 클로저 수명 = click 한 번. 끝나면 plaintext 변수 scope 종료.
- `MainWindow.LoadCliGroups` 복원 경로도 동일 패턴 — 탭 복원 시 한 번 복호화.
- `InitializeTerminal` 의 clipboard / autofill 워머 블록 **제거** (~30 라인).
  plink 가 native 처리하므로 더 이상 필요 없음. 코멘트로 "follow-up #3 에서
  지웠음" 명시.

### 보안 trade-off (v1 → v2 → v3)

| 비교 | v1 (clipboard) | v2 (PTY autofill) | v3 (plink) |
|---|---|---|---|
| Plaintext 디스크 잔존 | 없음 | 없음 | 없음 |
| Plaintext 메모리 잔존 | 클립보드 ~수분 | autofill object ~1 초 평균 (최악 2 분) | click 클로저 한 번 + ProcessStartInfo argv |
| Process listing 노출 | 없음 | 없음 | **연결 중 plink argv 에 보임** |
| 자동화 충실도 | 사용자 paste | OutputReceived race | **plink native 처리, race 없음** |
| OpenSSH 호환성 | 호환 | 호환 | 비호환 (plink 사용) |
| 추가 의존 | 없음 | 없음 | **plink.exe (PuTTY)** |

operator 가 명시적으로 v3 선택 (`"패스워드만 plink 적용"`) — process listing
노출은 single-user desktop 에선 수용 가능, 대신 자동화가 깔끔해지는 것에
가치를 둠. 디스크엔 여전히 DPAPI ciphertext 만.

## 후속 수정 #4 — plink 폐기, openssh + poll-based autofill 로 정착 (FINAL)

### Operator 보고 (verbatim — 5턴에 걸친 iteration)

> 'plink.exe'은(는) 내부 또는 외부 명령... 아 plink가 프로그램이였음? ssh 기본으로 다시 가죠..
> 반응이 없는데 터미널에 text전송후 엔터 전송하면 되는거아님? password 확인후 0.5초뒤
> 로그파악해 단순수정같은데 삽질을 너무하는것같음
> 이전에 뛰운창은 됨 신규로 뛰우면 안됨
> 다시 둘다 안됨
> 잘작동함 커밋하고 푸시

### 진단 (로그가 4번 검증해줌)

**A. plink 폐기**: operator 가 PuTTY 미설치 상태에서 `plink.exe is not recognized`
오류 받음. argv 로 비밀번호 보내려면 PuTTY 설치 필요한데, 단일 사용자
환경에선 그 비용보다 openssh + 자동 typing 이 더 간단. → plink branch 와
plaintext-at-click 클로저 전부 revert, openssh 로 회귀.

**B. autofill 부활 — 그러나 DispatcherTimer 스레드 버그**: v2 의 PTY 출력
watcher 가 race-prone 했던 진짜 원인 두 가지가 로그로 드러남:

1. **`GetConsoleText` 가 PTY stdout 이 아니라 Windows 콘솔 타이틀바 history
   를 반환**. v2.1 (이전 follow-up #2 에서 `ReadOutput` 으로 교체) 가 정답
   이었는데 operator 가 그 빌드를 테스트하기 전에 plink 로 피벗.
2. **`DispatcherTimer` 를 `OutputReceived` 핸들러 (ConPtyTerminalSession 의
   ThreadPool 폴링 스레드) 에서 생성**: ThreadPool 스레드의 auto-created
   dispatcher 는 message pump 가 안 돌아서 settle Timer 의 `Tick` 이 영영
   안 발화. operator 가 정확히 짚어준 케이스 — restored tab 은 작동
   (initial-scan 이 UI thread 에서 매칭됨), new tab 은 안 됨 (frame 경로
   = ThreadPool → DispatcherTimer 죽음).

   ```
   [Ssh-AF] frame#1 ... match=True text=<<<\e[?25l\e[Hpsmon@192.168.0.64's password:...
   [Ssh-AF] delivery scheduled ... delayMs=500
   (DELIVERED 영영 안 옴 — settle timer 가 ThreadPool 위에서 죽음)
   ```

3. **`OutputReceived` 자체가 발화 안 함 (frames=0)**: dispatcher 마샬링
   수정 후에도 `[Ssh-AF] EXPIRED ticks=741 frames=0 attached=True` 가
   여전히 나옴. 출력은 분명히 grow (471 → 502) 했는데 이벤트는 0회.
   원인은 ConPtyTerminalSession 의 50ms 폴링과 외부 구독자의
   관계에서 알 수 없는 race — 깊이 파보다 **이벤트 의존 자체를 폐기**
   하는 게 더 빠른 결정.

### 수정 (3단계로 점진적 단순화)

**1. revert plink → openssh** (SshCommandBuilder, MainWindow, dialog tip, 테스트)

`BuildSshCommand` / `ComposeArguments` 의 `plaintextPassword` 인자 제거,
Password 모드는 `ssh user@host` 만 emit. 테스트 invariant `argv 에 -pw /
password 등장 금지` 복원.

**2. `SshPasswordAutofill` 재도입** (event-driven, 단명)

`OutputReceived += ...` 패턴 + `ScheduleDelivery → DispatcherTimer` settle
delay. operator 가 dispatcher 스레드 버그 발견 후 UI dispatcher 로
`BeginInvoke` 추가. 하지만 OutputReceived 자체가 안 터지는 케이스 발견.

**3. 이벤트 폐기, 순수 폴링 (FINAL)**

```
DispatcherTimer 150ms (UI thread) 마다:
  - tab.Session 있나? 없으면 대기
  - session.OutputLength > _lastScannedLen ? (grow 확인)
  - 새 부분만 ReadOutput, _lastScannedLen 업데이트
  - needle "assword:" 매칭 → ScheduleDelivery (500ms settle) → WriteAndSubmit
  - 매 grow 마다 tail 160 자 로그 → 실제 ssh 출력 가시화
```

특징:
- **이벤트 구독 없음**: ConPtyTerminalSession 내부 이슈 영향 안 받음
- **UI thread 단일**: 스레드 race 없음, dispatcher 항상 정상
- **Incremental scan**: `_lastScannedLen` 으로 새 부분만 읽음, 큰 출력에서
  도 가벼움 (CPU + allocations 둘 다)
- **needle 경계 보호**: `scanStart = max(_lastScannedLen - needle.Length, 0)`
  로 needle 이 tick 경계 걸쳐도 매칭
- **모든 grow 로깅**: operator 가 매칭 안 되는 경우 실제 텍스트 한 줄로
  확인 가능 → needle 즉시 수정 가능

### 교훈 / Postmortem

- **WPF + ConPTY 의 thread 교차** 는 함정. `DispatcherTimer` 가 어느 스레
  드의 dispatcher 에 붙는지 항상 명시적으로 처리해야 함. `Application.
  Current.Dispatcher` 만 신뢰.
- **외부 이벤트 의존을 줄이고 polling 으로** 갈 때 코드가 더 단순해지는
  경우 — race + lifecycle 관리 복잡도가 polling 비용보다 훨씬 큰 상황에서
  특히 그렇다. 본 미션이 정확히 그 케이스.
- operator 의 **`"이전 창은 됨 신규로 뛰우면 안됨"`** 같은 차이 보고가
  결정적이었음 — 같은 코드 두 path 가 다르게 동작 = 환경 차이 (스레드
  context) 라는 가설이 즉시 떠올랐다.
