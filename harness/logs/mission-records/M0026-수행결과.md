# M0026 수행결과 — 에이전트 악단 유튜브 임베드 무대

## 실행 요약

미션 brief = Web플러그인 **에이전트 악단(agent-band)** 에 유튜브 임베드 무대를 추가.
- 무대 상단에 유튜브 링크 붙이기 → 메타정보 획득 → 카테고리화 → 재생목록 선택
- 영상은 스테이지 상단에서 재생, 하단의 악단·연주단은 그대로 유지(창 크기 변경 없음)
- **펜슬로 먼저 디자인 → 검토 → 구현** 게이트 명시

브리프대로 **2단계**로 진행했다.

### 1) 디자인 단계 (게이트)
- 사전 확인 2건: 메타/카테고리 = **oEmbed + 설정된 LLM 분류** 병행 / 악단 반응 = **시스템 루프백 그대로**.
- 펜슬 설계안 작성: `Docs/design/M0026-agent-band-youtube.pen` (앱 윈도우 목업 + 설계 노트 보드).
- 운영자 검토 → "의도와 맞음" 승인, LLM 호출은 **호스트 stateless op 신설** 채택.

### 2) 구현 단계
설계대로 호스트 2 op 신설 + 플러그인 UI 분할. **악단 반응 코드는 무변경** — 영상 소리가
시스템 믹서 → 기존 SystemLoopback 캡처 → AST tick 으로 그대로 흘러 연주자를 구동한다.

## 결과

### 호스트 측 (.NET) — 신규 브리지 op 2개

| 파일 | 변경 |
|---|---|
| `WebDevHost.cs` | `YouTubeOEmbedAsync(videoId)` — oEmbed 메타를 **호스트에서 fetch**(브라우저 CORS 우회). videoId를 `[A-Za-z0-9_-]{8..16}`로 검증 후 canonical `watch?v=`로 URL 재조립 → **SSRF 불가**. `ClassifyAsync(title, channel, categories)` — `LlmGateway.OpenSession()`을 `await using`으로 1회용 사용(= `SummarizeChunkAsync`와 동일 패턴) → **chat.* 히스토리 미오염**. 모델의 자유 응답을 화이트리스트로 클램프(`MatchCategory`), 미스매치 시 마지막 카테고리로 폴백. 정적 `HttpClient`(8s 타임아웃) + 레코드 `OEmbedResult`/`ClassifyResult` 추가. |
| `WebDevBridge.cs` | `youtube.oembed` / `llm.classify` 디스패치 케이스 2개 추가. JSON `categories` 배열 파싱 → `List<string>`. |
| `Wasm/common/zero-bridge.js` | 타입드 표면 `zero.youtube.oembed` / `zero.llm.classify` 추가(문서/미래 플러그인용). |

> IZeroBrowser 인터페이스는 **확장하지 않음** — `summarize`/`music.*`와 동일하게
> `WebDevHost` 메소드 + Bridge 케이스로만 구현(공식 표면 오염 방지).

### 플러그인 측 (web, `Project/Plugins/agent-band/`)

| 파일 | 변경 |
|---|---|
| `index.html` | `<main>` 재구성 — `#yt-bar`(무대 상단 URL 입력+붙이기+메타 상태) / `#yt-top`(영상 `<iframe>` + 캡션 + 재생목록 패널, 영상 없으면 `.hidden`) / `#band-region`(기존 canvas+overlay 이동). 헤더 hint·tag·title을 M0026으로 갱신. |

| `manifest.json` | `0.12.2 → 0.13.0` (minor — 신규 기능) + description 갱신. |

### 데이터 흐름 (구현된 대로)
```
[붙이기/Enter] → parseVideoId → iframe 즉시 재생(autoplay, 사용자 제스처)
   → youtube.oembed(host) → 제목·채널·썸네일 → 캡션·재생목록
   → llm.classify(host, stateless) → 카테고리 1개(실패 시 keyword→'기타')
   → 재생목록에 dedupe 저장 → 탭으로 필터·클릭 재생
악단: 영상 소리 → 시스템 믹서 → SystemLoopback 캡처 → AST tick → 연주자 (무변경)
```

## 평가 (3축)

### 1. 코드 안전성 — A−
- oEmbed: videoId 정규식 검증 + canonical URL 재조립으로 **SSRF 차단**. 8s 타임아웃, 실패해도 영상 재생은 계속(`warn`만).
- classify: 1회용 세션이라 chat 미오염. 모델 출력은 호스트 화이트리스트로 클램프 → 제목 prompt-injection이 와도 **허용 라벨 외 출력 불가**(최악=폴백). `_chatLock`으로 LLM 접근 직렬화.
- 플러그인: 제목/카테고리/썸네일 전부 `escapeHtml` 경유 → XSS 차단. `window.zero` 부재 시 typed/​invoke 모두 가드.
- 미해결(검토 포인트 #2): WebView2가 youtube.com `<iframe>`/네트워크를 CSP·가상호스트 정책으로 차단할 가능성 — **이 환경에서 실측 불가**(스모크 필요).

### 2. 아키텍처 정합성 — A
- 창 grid(56/1fr/96) 불변, 1fr 셀만 flex 분할 → "창 크기 변경 없음" 충족.
- 악단 오디오 경로 **신규 0** — 기존 루프백·AST tick 재사용(운영자 선택).
- 호스트 op는 `summarize`/`note.*`/`music.*`와 동일한 escape-hatch 패턴, IZeroBrowser 불변.

### 3. 테스트 가능성 — B+
- `parseVideoId`/`keywordCategory`/`MatchCategory`는 순수 함수 → 단위 검증 용이(현 라운드 미추가).
- iframe 임베드·oEmbed·LLM 분류는 통합/육안 검증 영역.

## 후속 수정 #1 — 장르가 모두 "기타"로 분류

### 운영자 보고 (verbatim)
> 잘 작동함.. 음악 두개를 붙였는데 음악 장르가 해석이 안되는데 llm 이용 해석한 부분 체크...
> 메타정보로 아마 인터넷을 한번검색후 정보획득후 카테고리를 분류하는 전략을 사용해야할것같음
> 현재는 모두 기타로 분류되는것으로보임

### 진단 (named root cause)
**LLM 백엔드 미로드** — 메타 부족이 아님.
- 앱 로그(`bin/.../logs/app-log.txt`): `[WebDev] oembed ok`는 있으나 **`classify` 라인 0개**, 모델 로드 라인 없음(Vulkan 디바이스 열거만 = 초기화). → `ClassifyAsync`가 `!LlmGateway.IsActiveAvailable()`에서 **로그 없이 `ok:false` 반환** → 플러그인 `keywordCategory` 폴백 → "BABYMONSTER‑DRIP / EJAE‑DNA" 제목이 regex에 없어 → 전부 **기타**.
- 운영자 제안(watch‑페이지로 메타 더 획득)은 **실측 반증**: 쿠키 없는 GET은 YouTube 동의/일반 페이지를 받아 `keywords`가 "동영상,공유,카메라폰…" 보일러플레이트만 옴(장르 없음). watch‑스크랩으로는 장르를 못 얻음 → 진짜 "검색" enrich는 별도 검색 API가 필요(검토 포인트로 분리).

### 수정 (저위험 3종 — 가시성·견고성)
- `WebDevHost.cs` — `ClassifyAsync`: ① not‑ready 경로를 **로그**로 남기고 에러 토큰을 `"llm-not-ready"`로 안정화(플러그인이 키 분기), ② 프롬프트 강화(가수·그룹 지식 활용 + few‑shot + 단일 라벨 강제).
- `index.html` — 캡션 배지에 `id="ytBy"` 부여.
- `agent-band.js` — `classifyVideo`가 `{category, by:'llm'|'keyword', llmOff}` 반환 → 캡션에 **출처 배지**(`✦ LLM 분류` vs `≈ 키워드 추정`) + 메타 상태에 "LLM 꺼짐 → 키워드 추정" 노출. `keywordCategory(title, channel)`로 채널 포함 + 아이돌/그룹·OST 어휘 보강(주제가 우선순위).
- `agent-band.css` — `.cap-llm.guess`(앰버) 추가.
- `manifest.json` — 0.13.0 → 0.13.1.

### 빌드 재검증
- 호스트 빌드 ✅ 오류 0. `node --check` ✅. 임시 zip 재생성: `%TEMP%\agent-band-v0.13.1.zip`.

### 핵심 결론 / 운영자 결정
- **운영자 결정: "LLM 켜서 사용"** (웹검색 enrich 미채택). 추가 호스트 개발 없음.

- 본 후속의 코드 변경(출처 배지·프롬프트 강화·not‑ready 로그·키워드 보강)은 LLM‑on/off 양쪽 모두에 유효하므로 그대로 유지.

## 후속 수정 #2 — 실제 설정 LLM을 쓰는 유닛테스트

### 실행 결과 (이 머신, 설정된 LLM 실사용)

## 후속 수정 #3 — 재생목록 영속화 + 걸그룹 정책 변경

### 운영자 요청 (verbatim)
> agent-band 는 다음과같이 한번더 개선... 한번 재생한 목록을 내부웹 스토리지에 저장해 유지해줄것 (다음에도 사용할수 있게, 재생목록을 내부에서 열고닫을수 있게도제공)... 그리고 현재 걸그롭에 메인보컬은 제외해죠.. 걸그룬이 아닌곳에 싱글인 설정인경우 메인여자 보컬로(이미구현됨,걸그룹정책만 변경).. 걸그룹일때 7인 설정 사용하면 됨..
